第一节 战略意义与项目背景
================================================================

一、项目概况

童慧行走天下是童慧行生态下的亲子旅行攻略子品牌。产品定位是面向广大家庭的亲子旅行规划工具，重点解决"亲子出行信息过载、攻略筛选耗时、缺乏儿童视角考量"这一家庭决策难题。

项目本身不是独立产品，而是已有童慧行生态中的一项功能扩展。具体形态包括两个核心组成部分：移动端应用程序内的走天下标签页，主站个人中心的旅行记录与勋章模块。两个组成部分共享同一套数据，共享同一套用户身份。

二、项目来源

项目起因为以下五项宏观背景：

第一，小红书母婴人群调研显示，超过百分之九十六的母婴用户以小红书作为日常了解母婴资讯主要渠道，每日打开小红书应用约二十二次，人均使用时长约四十一分钟，攻略收藏代替阅读是被验证的高频行为。这意味用户在攻略工具上的真实使用模式是碎片化、搜索驱动、收藏驱动的，不应设计成沉浸式阅读长文型产品。

第二，童慧行已有学习类应用验证了 uni-app 加 Vue3 加 JSON 编译期导入数据的产品形态。学诗词应用采用此架构支撑了两千零二十八首诗词的内容规模与流畅体验。走天下可以沿用同一套技术形态，降低研发成本与维护成本。

第三，童慧行主站用户中心已具备成就墙、个人积分、儿童管理等基础设施。走天下的勋章模块与记录模块可与现有用户中心组件无缝对接，复用程度高。

第四，童慧行已部署应用作为整体移动端产品已上线，五大标签页结构稳定。走天下作为新增标签页，无需重新构建整体移动端产品框架，仅作为原有壳子的新成员加入。

第五，旅游行业内，携程、马蜂窝等主要竞品均面向一般旅行者，对亲子场景缺乏深度支持。"孩子说好才是真的好"是童慧行的核心产品哲学，这一哲学在亲子旅行场景下有独特的差异化价值。

三、项目目标

本方案目标是按以下顺序实现三项目标：

第一目标，按育儿场景完成数据底座建设。首批支持十大热门亲子旅行目的地，约五十至八十个高质量亲子景点，三十类亲子玩法，五至八种标准行程模板。

第二目标，完成移动端原生页面的功能闭环。用户可以浏览亲子目的地，可以读取攻略详情，可以按四步向导生成定制行程，可以收藏攻略，可以查看个人偏好与本地草稿。

第三目标，完成主站扩展。用户登录主站个人中心可见自己的旅行总览，包含累计次数、天数、收藏攻略数、家庭最爱城市等关键指标，并可获取走天下勋章。

四、非目标

以下内容不在本方案范围内：

不做独立应用商店上架，不在子品牌层做应用分发，依靠童慧行移动端应用分发。

不做旅游产品交易。本方案不实现导游预订、景点票务、保险购买等交易行为。这部分留到 v2 立项。

不做社交内容生成。本方案不实现攻略发布、评论、收藏等创作者经济功能。本阶段只读、不写社区。

不做跨语言版本。v1 仅支持简体中文，后续可扩展。

五、与已有产品关系

童慧行已有产品矩阵包括：主站、学诗词、学国学、学通识、学英语、来挑战、童慧行论坛、积分商城。各模块关系如下：

走天下不与学习类应用共享数据。学习类应用重在儿童独立学习，走天下重在亲子互动场景。两者目标人群有部分重叠（儿童本身），但决策者重叠度更高的是家长。

走天下与论坛存在远期协同可能。论坛上家庭日常分享可与走天下攻略形成互补。但 v1 阶段两模块不联动。

走天下与积分商城存在潜在兑换可能。完成旅行记录、获得勋章等行为可兑换商城物品（v2 阶段评估）。

走天下与主站用户中心是本期方案中重点联动对象。详见第七节。

第二节 用户定位与价值主张
================================================================

一、用户优先级核心定性

童慧行走天下的产品定位核心是：

真正的用户始终是父母（家长），但服务的对象是孩子。父亲母亲不是为消费体验买单，是为孩子体验买单。

在每次亲子出行场景下，用户优先级排序为：

第一位，孩子体验。是这次出行的核心评估人，决定这次出行值不值。孩子说好才是真的好。

第二位，母亲体验。是主要决策者与主要照护者，决定去不去、买不买。母亲会作为攻略阅读与评估的主力人群。

第三位，父亲体验。是行动者与买单人，决定怎么去、订哪个。父亲在攻略消费链中处于决策链下游。

二、用户优先级对产品的影响

这一优先级贯穿到产品每一个细节。

第一，景点详情的展示结构必须是孩子亮点优先，母亲亮点其次，父亲亮点最后。kidHighlights 字段为必填且至少三项。

第二，行程节奏必须包含午休窗口与晚间哄睡窗口。母亲最在意的是孩子体力与作息，父亲很少因睡眠问题争论。

第三，餐厅筛选必须包含儿童椅、是否有吵闹包间等家庭友好指标。小资文艺型推荐不作为主要推荐，父亲单一取向的成人场景不主推。

第四，城市介绍的开篇必须是孩子钩子（kidHook），母亲钩子（momHook）放中部，父亲钩子（dadHook）放最末。

第五，首页视觉要做到母亲审美（高级、柔和、留白）与孩子钩子（儿童化大字、卡通图标）的反差共存。前者让母亲愿意停留，后者让孩子愿意点入。

三、目标用户画像

年龄分布：核心二十五至三十五岁，覆盖至四十五岁。一线及新一线城市为主。

家庭结构：核心家庭（三口之家）+ 三代同堂家庭（祖父母参与）+ 单亲家庭（带一孩）。

收入水平：中产阶层。每月可支配旅行预算三千至八千元。

技能水平：会用智能手机，使用过小红书，对旅行攻略有阅读习惯。

决策周期：典型决策链为两周到两个月。决策期内会多次打开多个 App 比对。

四、价值主张一句话

为有孩家庭提供以孩子体验为核心、母亲决策为导向、父亲行动为闭环的亲子旅行规划工具，让每一次亲子出行决策更轻、更准、更稳。

五、价值维度拆解

本项目在以下五个维度为用户创造价值：

功能价值：把原本散落各 OTA、社交平台的亲子旅行信息聚合到一起，提供结构化、亲子化的标准化数据。

情感价值：让孩子参与决策时能看到"为孩子量身定制"的内容，强化家庭情感连接。

社会价值：用户使用产品获得勋章、积累记录，可在妈妈圈、亲友群中获得展示资本。

经济价值：节省母亲四处查攻略的时间成本，避免父亲决策失误的返工成本。

知识价值：把"亲子旅行怎么做"的专业知识体系沉淀到产品中，让普通家长也能做专业攻略。

六、价值证据

每一项目标价值如何衡量：

功能价值：攻略完成率（用户完成四步向导比例）、收藏率、停留时长。

情感价值：主站勋章获得率、月活跃家庭数（重复访问 2 次以上）。

社会价值：勋章展示次数（来自主站晒图）、攻略分享数。

经济价值：用户复访率（三十日复访）、NPS 评分。

知识价值：用户在攻略详情页平均获取信息点数量、攻略分享二次传播率。

具体量化指标见第十一节。

第三节 数据模型
================================================================

一、模型总体结构

走天下 v1 数据模型由以下五张主数据表与若干派生索引构成：

城市表（City）
景点表（Spot）
餐厅表（Restaurant，可与 Spot 合并到同一基础类，后续讨论）
亲子玩法表（Activity）
行程模板表（ItineraryTemplate）
季节提示表（WeatherTip）

不进入 v1 的表：旅行社表（Agency）、导游表（Guide）、咨询订单表（Inquiry）。v2 评估。

二、城市表字段

城市表存储静态城市信息。

必填字段：城市 ID、城市拼音名、城市中文名、行政区划、所属大区（华北/华东/华南/西南/西北等）、封面图 URL、适合出行月份数组、平均气温（春/夏/秋/冬四项）、城市标签数组。

亲子化字段：kidHook 一句话城市钩子（孩子向）、momHook 一句话城市钩子（母亲向）、dadHook 一句话城市钩子（父亲向）。

元字段：城市简介（150 字以内）、行政区号、坐标经纬度、数据采集时间、采集人 ID。

三、景点表字段

景点表为最丰富数据表，是攻略详情页内容主要来源。

基础字段：景点 ID、所属城市 ID、景点中文名、拼音名、景点类型枚举（自然/主题乐园/博物馆/动物园/海滩/古城/古城/古街/购物/演艺等十六类）、分类标签数组。

亲子字段：推荐月龄段（最小年龄、最大年龄）、孩子亮点数组、母亲亮点数组、父亲亮点数组、孩子痛点数组、母亲痛点数组。

运营字段：营业时间段、推荐游玩时长（小时）、成人参考票价（元）、儿童票价（元，可空）、几岁以下免费（可空）、推车便利度三档（良好/一般/不佳）、母婴室有无、是否需要预订（必须/可选/不需要）、最佳游玩时段数组、是否建议雨天避免、交通方式说明、周边亲子餐 ID 数组。

评分字段：总体评分一星至五星、孩子评分、母亲评分、父亲评分。

展示字段：星级分布、用户评价数量、封面图 URL、画廊图片数组、避坑提示数组。

四、亲子玩法表字段

亲子玩法表描述在某个景点内可执行的活动形式，是骨架叠加时的关键素材。

必填字段：玩法 ID、玩法中文名、玩法类型枚举（沙水/动物/演艺/手作/骑行/博物馆/雪上/水上/攀爬等）、推荐月龄段、体力消耗三档（低/中/高）、单次持续时长（分钟）、是否需要推车、是否需要安排在午休时段。

可选字段：装备清单数组、玩法说明文本、危险度评级、是否可重复体验。

五、行程模板表字段

行程模板表定义可复用的行程骨架。

字段：模板 ID、模板名（如"北京 5 天 4 晚 三代同堂 + 紫禁城"）、覆盖天数、适用季节枚举或枚举数组（春/夏/秋/冬/任何）、按日骨架数组。

按日骨架中每一天包含：该日天数索引、该日主题（如"抵达与适应"）、时间块数组（上午/中午/下午/傍晚/晚间）、每块标记类型（景点/餐厅/休息/酒店/自由）、对应的景点 ID、备注。

亲子评估字段：孩子友好度评分（1-5）、母亲友好度评分。

六、季节提示表字段

季节提示表是按城市、按季节的轻量内容，主要服务于首页"何时去"的卡片文案与目的地的可读性。

字段：所属城市 ID、季节枚举、孩子出行提示、母亲出行提示、父亲出行提示。

七、字段命名规范

所有数据库字段采用驼峰命名。TS 接口定义采用相同驼峰命名。JSON 序列化时保持原驼峰结构，不转换为下划线。

枚举值采用字符串，不使用数字代号，便于跨语言读懂。

所有评分字段使用 1-5 整数星级。所有时长字段使用小时或分钟单位。所有金额字段使用整数元。

八、索引与性能

v1 数据规模较小（景点不超过五百条），可整体编译到 App 包内，无需冷加载。本期不设置服务端索引。

主站扩展勋章系统需要服务端查询时，主站现有数据库结构已具备相应字段扩展能力，详见第七节。

九、与现有数据规范的关系

走天下沿用童慧行已有数据驱动应用的命名习惯：

参考 apps/xueshici/src/data/poems-data.json 的结构组织：顶层数组、每条记录核心 ID、嵌套对象表达复合结构。

不引入超过三层嵌套，避免前端代码复杂度膨胀。

字段定义、类型、约束条件、必填规则、示例值等全部细节见附录 A。

第四节 应用架构与页面结构
================================================================

一、应用架构

走天下作为童慧行移动端应用的第五个标签页，与其他四个标签共享同一套移动端壳子。

移动端技术栈：uni-app + Vue3 + TypeScript + Pinia。

页面形态：原生 Vue 单文件组件（不嵌套 WebView）。

数据形态：JSON 编译期导入，前端处理。

主要遵守规范：童慧行已有规范（参考学习类应用代码风格）。

二、页面清单

走天下 v1 涉及以下七个一级页面与若干二级页面：

一级页面一：走天下首页（原生 Vue 页面）

二级页面：城市详情页、攻略详情页、我的草稿列表页、我的收藏列表页、搜索结果页

一级页面二：生成攻略四步向导（入口在首页中部与底部）

二级页面：第一步选择目的地、第二步设置出行条件、第三步设置偏好、第四步确认大纲

一级页面三：调整版（输入已有攻略 ID 后生成定制版）

一级页面四：勋章展示（v1 可选，访问入口在主站个人中心）

三、首页结构

首页自上而下分以下五块：

第一块：顶部搜索框

第二块：编辑精选·热门亲子目的地（横滑卡片）

第三块：四步生成攻略入口（强势 CTA）

第四块：本周热门城市列表（竖向折叠列表）

第五块：用户偏好与草稿入口（已登录用户可见）

四、城市详情页结构

城市详情页自上而下分以下五块：

第一块：城市封面图与基本信息

第二块：孩子钩子与母亲钩子与父亲钩子并列（视觉反差点）

第三块：可游景点列表（按孩子评分排序）

第四块：亲子玩法类型筛选器

第五块：季节提示卡片

五、攻略详情页四件套

攻略详情页是核心页面，采用以下标准结构：

顶部第一件套：孩子钩子（kidHook 一句话 + 孩子入镜图 + kidRating 5 星评分）。

中段第二件套：花费清单（成人/儿童/家庭总计），明码标价。

中部第三件套：时间线（含午休与哄睡窗口图标）。

底部第四件套：避坑三条（红色框醒目）。

底部固定 CTA：收藏按钮、生成我的版本按钮。

六、四步向导结构

第一步：选择目的地。可输入城市或选择首页推荐城市。

第二步：设置出行条件。包含天数、人数、儿童年龄、季节。

第三步：设置偏好。包含景点类型多选、交通方式多选、每日预算、住宿类型。

第四步：调出拼装引擎返回的大纲，用户可手动调整。确认后进入生成模式。

七、入口与导航调整

童慧行移动端应用已有 5 个 Tab（首页、学习、挑战、走天下、我的）。走天下作为其中之一，无需调整 Tab 数量。

首页（Vue 入口）需要为走天下 tab 添加 banner 与入口卡，确保走天下 tab 被新用户接触率达标。v1 阶段以"插入式 banner"为主，不改动首页整体结构。

八、与主站的导航关系

走天下在主站的入口是"个人中心"的走天下记录模块。用户访问个人中心后，看到累计旅行次数、勋章列表等数据。

反向导航：从走天下回到主站，使用 deep link（如 grandand://personal-center）。

九、跨端编译策略

走天下页面与童慧行其他学习类页面共用同一套 vue3 + uni-app + ts 工程，一键产出 H5 与 iOS 与 Android 与小程序。

H5 渠道：用户通过 grandand.com/mobile 路径访问。

小程序渠道：首期不支持。v2 评估。

原生 App 渠道：iOS App Store + Android 国内主流应用市场。

第五节 拼装引擎算法
================================================================

一、引擎目标

拼装引擎（Assembler）走天下产品的核心技术组件。引擎目标是把用户四步输入参数转化为一份结构化的行程大纲，可直接渲染为可读攻略。

二、引擎定位

引擎做的是"按规则筛选与拼装"，**不**依赖任何大语言模型。整个引擎是一个纯函数（输入输出确定、可解释）。

三、输入参数

用户四步向导的全部产出作为输入：

出发地（字符串）
目的地（字符串）
出行天数（1-30 整数）
出行人数（含儿童）
儿童年龄数组（如 [3, 5]）
出行季节（春/夏/秋/冬）
景点类型偏好数组
交通方式偏好数组
每日预算（元）
住宿偏好（文本可选）

四、输出结构

引擎输出一个完整行程大纲对象。包含：

总览标题、概述文本、行程天数、按日数组。

每日包含：

该日主题（如"抵达与适应"、"主题乐园日"、"城市探索日"）、该日时间段块数组（上午/中午/下午/傍晚/晚间）、每块标记（景点/餐厅/休息/酒店/自由）、每块对应的具体景点 ID 与餐厅 ID、每块的备注（如午休或用餐提醒）。

五、引擎实现步骤

引擎由五步流水线组成：

第一步，模板选择。基于目的地、季节、天数，从 ItineraryTemplate 表匹配最佳模板。匹配策略是按目的地优先级最高，其次按天数差异最小，最后按季节匹配。

第二步，景点筛选。从模板骨架出发，按用户儿童年龄数组取交集筛选景点，要求每个景点推荐月龄段覆盖用户儿童年龄。

第三步，填补与替换。模板骨架中的每个景点 ID 替换为已筛选的具体景点。若筛选结果不足，按孩子评分降序补足。

第四步，节奏调整。根据景点推荐游玩时长，校验每天时间块是否过满。过满则降低下午某景点时长或舍弃某景点。午休与晚间哄睡窗口强制保留。

第五步，输出对象构造。补齐每块备注、生成总览文本，输出大纲。

六、规则化评分

每个景点都有三个评分：孩子评分、母亲评分、父亲评分。筛选评分阈值分别为：孩子评分至少 4 星、母亲评分 3.5 星以上。

排序：默认按孩子评分降序。预算偏低时按"家庭喜爱度加权"。

七、避坑提示抽取

引擎不生成文本，但会从每个景点的"避坑提示"数组抽取并按主题归类（基础设施/天气/餐饮/突发），输出到尾部"出行注意"卡片。

八、引擎性能

v1 数据规模（十条线路、五十至一百景点），单次引擎运行预计耗时 < 50 毫秒。

引擎代码量预估约 300 至 500 行 TypeScript。

九、引擎与 UI 的边界

引擎只产出结构化大纲，不产出散文。UI 渲染层负责把大纲转为人可读的卡片视图。

引擎不调用任何外部 API，无网络依赖。

十、不做的 AI 行为

明确以下三类行为不在引擎能力范围内：

不基于历史偏好推理（v1 用户偏好只作为输入，不参与模型推理）。

不解释攻略内容（攻略详情页文案均为策划手工撰写）。

不与客服对话交互（无对话能力）。

这些行为若未来需要，需立项专项开发。

第六节 用户界面设计原则
================================================================

一、设计哲学

走天下产品的 UI 设计原则来源于在小红书母婴用户行为上的关键调研结论。

二、小红书母婴用户行为关键数据

数据一：母婴用户每天打开小红书约 22 次，每次停留约 41 分钟。说明用户使用方式是碎片化反复浏览，不是一次性深度阅读。

数据二：每天搜索约 8 次。说明搜索行为是用户主要输入路径，而非推荐流。

数据三：每日小红书搜索 3 亿次，母婴搜索同比 +60%。说明攻略消费高度依赖搜索入口。

数据四：母婴用户 96% 选择小红书作为日常了解母婴资讯主要渠道。说明小红书是攻略消费的标杆平台。

数据五：母音用户主导决策与消费。说明母亲是阅读攻略的主要受众。

数据六：5-7 次触达才转化。说明攻略工具要让用户能收藏、复访，反复接近才转化。

三、UI 必做原则

原则一：卡片信息密集

每张攻略卡片标题必须按以下格式组织：

方括号【目的地 · X 天 X 晚 · X 岁娃 · 出行方式】人均金额元 + N 个避坑

第二行副标题带孩子兴奋点（与具体景点相关）

原则二：首页横滑卡

首页首屏必备横滑卡片展示十大热门目的地。每张卡是 16:9 图片、目的地名、kidHook 一句话、推荐游玩季节。

原则三：详情页四件套

攻略详情页头部固定按以下四块顺序展示，不允许前后顺序：

第一件套：孩子钩子（kidHook + 孩子入镜图 + 孩子评分 5 星）。

第二件套：花费清单（成人价、儿童价、家庭总计，明码标价）。

第三件套：时间线（含午休图标与哄睡窗口图标）。

第四件套：避坑三条（红色框醒目）。

原则四：底部浮动 CTA

攻略详情页底部固定两个按钮：收藏、调整生成我的版本。两个按钮常驻。

原则五：搜索框置顶

首页顶端有搜索框。点击进入城市搜索、景点搜索两种维度的结果页。

原则六：视觉反差共存

主色 #16a34a（绿色），母亲审美风格，整体高级、留白。但首页首屏的关键区域用儿童化大字、卡通图标。视觉反差共存是核心要求。

四、UI 反面清单（绝对禁止）

以下行为是小红书母婴用户最反感的行为，走天下同样不应出现。

禁止一：假大空标题（如"最全亲子攻略"、"完美亲子游"）。

禁止二：无孩子入镜图，全是大景风景图。

禁止三：价格不透明（"性价比超高"这类描述）。

禁止四：长篇软文不带时间线与具体数字。

禁止五：强行插入无关电商链接。

禁止六：儿童营销腔（"宝宝成长必备"）。

五、视觉规范

主色：#16a34a（绿色）。

辅助色：#facc15（孩子黄色）、#ec4899（粉，母亲向）。

字体：母亲向标题用现代化无衬线（如 system-ui）；孩子钩子用圆润手写体。

图标：统一使用 SVG 矢量图形，不使用 emoji。走路天下严禁使用 :tada: 或 :rocket: 类表情替代图标。

按钮：圆角矩形，hover 高亮含即时反馈。

六、跨端适配

走天下页面在 iOS、Android、小程序三个端口渲染一致。

底部安全区域适配 SafeArea。

暗色模式适配：v1 仅浅色模式，v2 评估暗色。

七、交互细节

攻略卡点击进入详情页，无需二次确认。

收藏操作有即时反馈（toast 提示）。

四步向导之间过渡使用左右滑动动画。

不支持手势返回（uni-app 默认开启），需要在关键页面禁用。

八、设计交付物

走天下 v1 设计交付物包括：

首页 UI 设计稿（Figma）
城市详情页设计稿
攻略详情页设计稿（四件套固定）
四步向导设计稿
搜索结果页设计稿

第七节 主站个人中心扩展
================================================================

一、扩展背景

童慧行主站已有个人中心组件 PersonalCenter.vue，包含学习报告、儿童管理、青少年模式等功能模块。本方案新增"走天下记录"模块，与学习报告并列展示。

二、新增模块结构

走天下记录模块包含以下三个组成部分：

第一部分：走天下总览卡

第二部分：旅行明细列表

第三部分：勋章列表

三、走天下总览卡结构

总览卡位于"学习报告"模块下方，展示以下字段：

总旅行次数（用户所有孩子参与的旅行次数）
累计旅行天数
收藏攻略数
生效中攻略数（已调整生成我的版本）
家庭最爱城市 Top 3（按旅行次数排名）
最近一次旅行时间

四、旅行明细列表

旅行明细列表按时间倒序展示每次旅行的详情：

旅行主题
目的地（城市 + 核心景点）
参与儿童（昵称 + 年龄）
出行天数
出行时间（起始日期）
本次花费（元）
本次勋章（如有）

五、勋章列表

勋章列表展示用户已获得的走天下系列勋章。每枚勋章展示勋章图标、勋章名、获得时间、获得条件说明。

勋章总览指标：累计获得数、距离下一枚勋章进度。

六、个人中心集成方式

在 PersonalCenter.vue 中新增"走天下" Tab，位置安排在学习报告 Tab 之后、青少年模式 Tab 之前。

视觉上保持与学习报告同等的卡片样式。

七、数据获取

走天下记录数据通过以下接口获取：

接口一：GET /api/user/travel-stats
返回：累计旅行次数、累计天数、收藏攻略数、生效中攻略数、最近一次旅行时间、家庭最爱城市 Top 3。

接口二：GET /api/user/travel-records
返回：旅行明细列表，支持分页与时间范围筛选。

接口三：GET /api/user/travel-badges
返回：用户已获得的走天下勋章列表。

八、数据存储设计

主站数据库扩展方案如下：

User 表新增字段 travelStats JSON（家庭旅行统计聚合）。

新增 TravelRecord 表（旅行明细）。

新增 TravelBadge 表（勋章发放记录）。

新增 TravelBadgeDef 表（勋章定义）。

详细字段定义见附录 C。

九、勋章联动

勋章发放时机：

用户在 App 内调整生成"我的版本"并保存时，自动记录一条 TravelRecord。

TravelRecord 满足勋章发放条件时，自动为对应 TravelBadge 记录。

App 内打开主站时，通过 deep link 跳转到个人中心走天下记录模块。

十、设计原则

走天下记录模块设计原则与其他模块对齐：

卡片样式统一。
关键数据用大字突出。
勋章列表用统一 AchievementWall 组件。
总览数字以使用户感到"我还可以去更多"为目标，不以"我没去多少"为目标。

第八节 勋章体系设计
================================================================

一、勋章定位

勋章是用户通过走天下产生的行为证明，目的是让用户产生"再解锁一枚"的欲望，同时让其在主站个人中心有可展示资本。

二、勋章分类

v1 首批勋章全部按亲子场景设计。共五类：

类型一：城市探索者

类型二：三代同游

类型三：周末游侠

类型四：自然/海洋类（具体勋章名按城市主题命名）

类型五：人文/历史类（具体勋章名按城市主题命名）

三、城市探索者勋章

定义：首次解锁一个新城市即为该城市的探索者。

获得条件：调整生成该城市的"我的版本"并保存为旅行记录。

每城单独计数。同座城市首次解锁发一枚勋章，后续不重复发。

解锁城市勋章不计入勋章累计数（仅作展示用）。用户主勋章累计按"解锁过多少个城市"统计。

四、三代同游勋章

定义：与祖父母共同出行的旅行记录累计 1 次即可获得。

获得条件：旅行记录中标注参加人员中包含祖父母一代（年龄段 50+）。

解锁后单枚计数。

五、周末游侠勋章

定义：三个自然月内完成 4 次以上单天至两天短途旅行即可获得。

获得条件：旅行明细时间范围 90 天以内、单次天数 ≤ 2 的旅行记录数 ≥ 4。

未达成时显示进度提示。已达成后单枚计数。

六、按主题分类勋章

这一类勋章以"亲子主题体验"为核心，命名贴切具体场景，不与城市 1:1 绑定。

首批四枚：

勋章一，海岛小水手：在三亚解锁 3 次以上与海相关的玩法记录即可获得。

勋章二，熊猫粉丝：与儿童在成都完成熊猫主题行程至少 1 次即可获得。

勋章三，熊猫粉丝：在成都主题乐园行程完成一次解锁。

勋章四，古城探秘：在西安/北京/南京任意一座古城解锁 ≥ 2 次即可获得。

七、勋章定义存储

所有勋章定义存储于 TravelBadgeDef 表。每枚勋章包含以下字段：

勋章 ID
勋章名
勋章图标 URL
勋章描述
勋章分类
发放条件描述（用于 UI 展示）
发放条件结构化（JSON 字段，便于 App 端校验）
是否可重复获得（否，单枚即终态）

八、勋章发放机制

勋章发放时机有两个：

时机一：用户在 App 内调整生成"我的版本"并保存，App 调用 POST /api/travel/badges/check 接口，服务端校验是否满足任何勋章条件。

时机二：用户在主站个人中心查看旅行明细，手动补充家庭成员信息后，后端异步校验。

发放成功后，App 端实时通知用户；主站端下次访问时同步显示。

九、勋章展示策略

主站 App 顶部消息条：勋章获得即时提示。

主站个人中心：勋章列表按分类折叠。

App 侧：勋章在走天下 Tab 首页显要位置做一次性横幅庆祝，之后归入历史勋章。

十、勋章与推荐的关系

用户已获得的勋章可影响首页推荐策略。例如用户已得"熊猫粉丝"，首页可推成都相关新景点或新玩法。

v1 推荐策略较简单，仅在首页"为你推荐"区块展示该勋章对应城市后续可能感兴趣的内容。

v2 评估更复杂的勋章 + 推荐交互。

第九节 用户统一体系
================================================================

一、统一背景

童慧行已具备统一登录体系。所有模块共享同一套用户身份。本方案坚决遵守该约束，不引入新的账号体系。

二、统一要点

要点一：所有走天下相关页面与接口均依赖 grandkidsgo_token 验证身份。

要点二：本地存储 key 以 grandkidsgo_travel_ 为前缀，避免与现有 key 冲突。

要点三：走天下不提供单独的注册、单独密码、单独第三方登录。

要点四：已登录走天下等同于已登录童慧行，反之亦然。

要点五：用户注销走天下时（即退出 App 登录态），其在走天下产生的本地数据由 App 自动清除，服务端数据保留在主站。

三、登录工具复用

走天下前端直接调用 apps/mobile/src/pages/mine/auth.ts 的 getToken / getUser / isLoggedIn 等方法，无需二次封装。

四、本地偏好存储

用户在向导中选择的目的地、天数、孩子年龄等偏好信息，保存在本地，key 命名如下：

grandkidsgo_travel_profile：用户旅行偏好（JSON）。
grandkidsgo_travel_drafts：用户草稿列表（数组）。
grandkidsgo_travel_collect_<id>：单个收藏项。

五、服务端同步策略

v1 不做服务端同步。所有数据本地保留。

v2 评估服务端同步，可能引入独立的 TravelCloudSync 服务，复用 auth-service 的 JWT，不新增认证服务。

六、主站账号联动

用户在走天下 App 端登录后，访问主站 grandand.com/personal-center 时，自动通过 cookie 同步登录态。

反向导航通过 deep link 实现，从主站跳回走天下 App。

七、用户统一体系原则

原则一：单一身份源。童慧行只有一套用户身份。

原则二：单一登录入口。所有模块共用 auth-service。

原则三：单一 JWT。多端共用同一 JWT 进行身份验证。

原则四：本地与服务端解耦。本地数据可被清除，服务端数据保留作为权威记录。

第十节 实施阶段与工时估算
================================================================

一、总览

走天下项目预计总投入约 200-260 人小时，可拆分为以下五个阶段：

阶段 0 (MVP)：约 30 小时
阶段 1 (v1.0)：约 100 小时
阶段 2 (主站扩展)：约 60 小时
阶段 3 (v2 评估)：暂时不计划

二、阶段 0 MVP

目标：用最小可验证方式验证用户对走天下的核心需求。

内容范围：

第一批数据：1 城（北京）约 8 景点 + 5 玩法 + 2 模板。

原生 Vue 页面：走天下 tab 原生化（替换 webview），展示首页与城市详情页。

不包含：四步向导、收藏、搜索、主站联动。

工时：约 30 小时，含 1 人 × 4 个工作日。

验收：

走天下 tab 点击后能进入原生页面。
首页可见北京城市横滑卡。
点击北京进入城市详情页可见 8 景点列表。
本地构建可通过，模拟器可跑。

三、阶段 1 v1.0

目标：完成走天下 App 端完整功能闭环。

内容范围：

数据：扩展至 10 城 50+ 景点。

四步向导 UI 与拼装引擎。

攻略详情页四件套。

本地收藏与草稿。

搜索（城市 + 景点）。

埋点接入。

工时：约 100 小时，含 1 人 × 12 个工作日。

验收：

用户能完成从启动 App 到生成攻略的全流程。
攻略详情页四件套完整。

第十一节 度量指标与埋点方案
================================================================

一、目标

通过埋点验证产品假设，并基于数据驱动产品迭代。

二、关键指标

指标一：攻略完成率 = 完成四步向导的次数 / 启动四步向导的次数。目标 >= 50%。

指标二：卡片标题吸引力 = 攻略详情页停留 < 30 秒的占比。目标 <= 30%。

指标三：详情页深度体验 = 攻略详情页停留 >= 2 分钟的占比。目标 >= 40%。

指标四：收藏率 = 攻略详情页点收藏的次数 / 攻略详情页打开次数。目标 >= 8%。

指标五：四步向导每步转化率。每步之间流失率 < 15%。

指标六：城市横滑卡点击率 = 城市横滑卡点击数 / 城市横滑卡展示数。目标 >= 25%。

指标七：主站月活家庭数 = 主站个人中心走天下记录被访问的家庭数（去重）。首月目标 100 家庭 / 千 App DAU。

指标八：勋章获得率 = 已获得至少一枚勋章的用户数 / 登录用户总数。首月目标 >= 5%。

三、埋点事件

埋点采用 PostHog SDK（沿用童慧行已有方案）。

事件清单见附录 D。

四、埋点原则

原则一：埋点必须经过产品评审通过后才能上线。

原则二：埋点不收集用户敏感信息（孩子真实姓名、家庭住址）。

原则三：埋点 data 用 JSON 结构，便于后续扩展。

五、数据使用

数据用于：

产品迭代决策（哪些 UI 不及预期）。

A/B 测试（卡片标题改版前后对比）。

运营回顾（每周 metrics review）。

不用于：

机器学习模型训练（本项目无 ML 组件）。

第三方共享。

六、隐私合规

埋点需符合现有隐私合规标准。

走天下记录涉及孩子画像数据，需在用户首次启用时明示同意范围。

退出登录后即清除本地偏好与草稿，服务端数据保留按隐私政策执行。

第十二节 风险与决策记录
================================================================

一、风险清单

风险一：孩子评分主观性风险

说明：景点评分由采集人员主观打分，可能存在偏差。

缓解：每次采集后由另一位采集者独立复核。每月抽样回访用户校正。

风险二：数据规模扩张风险

说明：v1 后期数据量扩大后，编译期导入方式可能导致包体积过大。

缓解：v2 评估服务端 + 增量下载模式。

风险三：小红书 UI 趋势漂移风险

说明：小红书母婴人群 UI 趋势可能变化。

缓解：季度做一次用户调研 + UI 对比分析。

风险四：拼装引擎拼装质量风险

说明：算法拼装的攻略可能存在"模板感重"问题。

缓解：增加"灵活点"机制，允许用户调整每块时长与备注。

风险五：童慧行主站压力风险

说明：主站个人中心新增走天下模块可能对主站性能造成压力。

缓解：旅行明细接口设计为分页 + 缓存。

二、决策记录

决策一：走天下不是独立应用，作为 apps/mobile 一个 tab。

决策二：数据采用 JSON 编译期导入，不依赖后端服务（v1）。

决策三：不依赖大语言模型，纯规则引擎 + 模板拼装。

决策四：用户身份 100% 复用 grandkidsgo_token，不新建账号体系。

决策五：勋章仅按亲子场景设计。

决策六：主站走天下记录数据存储于扩展 auth-service 表，不另建数据库。

三、未决问题

问题一：勋章图标是否采用 SVG？

倾向：是，原则 4.4 拒绝 emoji。

问题二：是否需要支持"已登录但未填家庭成员"的状态？

倾向：需要，最小化默认填"家长本人"。

问题三：孩子昵称需要上传头像吗？

倾向：v1 不强制，复用现有孩子管理头像。

四、复盘节奏

每个阶段结束后做一次复盘，复盘内容更新入本方案第十二节。
