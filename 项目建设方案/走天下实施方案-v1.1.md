第一节 战略意义与项目背景
================================================================

一、项目概况

童慧行走天下是童慧行生态下的亲子旅行攻略子品牌。产品定位是面向广大家庭的亲子旅行规划工具，重点解决"亲子出行信息过载、攻略筛选耗时、缺乏儿童视角考量"这一家庭决策难题。

v1.1 起，走天下不再只是"用户填写四步参数 → 系统机械拼装一份行程"，而是引入 AI 对话式向导与已收录的结构化数据底座协同，提供两种互补的用户交互形态：
其一为对话引导模式，用户像与经验丰富的闺蜜聊天一样逐步描述诉求，由 AI 实时转写为结构化偏好；
其二为懒人一键模式，用户只填最少核心字段，AI 根据目的地画像与用户历史偏好智能补全其余字段。

两种模式最终汇入同一套拼装引擎，输出一致的结构化行程大纲，再由 AI 以三级口吻（孩子向、母亲向、父亲向）改写为人读自然语言。两条路径在数据底座、引擎算法、文案风格、生成约束上完全等价。

二、项目来源（沿用 v1.0 五项宏观背景，本节不重复）

三、项目目标（修订）

第一目标，按育儿场景完成数据底座建设。首批支持十大热门亲子旅行目的地，约五十至八十个高质量亲子景点，三十类亲子玩法，五至八种标准行程模板，并新增餐厅、酒店、交通、医院、商场、用户评价等真实数据来源。

第二目标，完成移动端原生页面的功能闭环。用户可以浏览亲子目的地，可以读取攻略详情，可以按对话引导模式生成定制行程，也可以按懒人一键模式生成，可以收藏攻略，可以查看个人偏好与本地草稿。

第三目标，完成主站扩展与 AI 能力建设。用户登录主站个人中心可见自己的旅行总览，包含累计次数、天数、收藏攻略数、家庭最爱城市等关键指标，并可获取走天下勋章。同时后台具备多供应商 AI 调用能力、智能反问阈值策略、三级口吻改写能力。

四、非目标（沿用 v1.0，本节不重复）

五、与已有产品关系（沿用 v1.0，本节不重复）

六、与 v1.0 的关键变化摘要

变化一，新增 AI 对话模式。v1.0 的四步表单作为懒人一键模式的载体保留，并行增加对话模式作为新默认入口。

变化二，拼装引擎升级为多套候选骨架输出。在 v1.0 的"按规则筛选与拼装"基础上，输出紧凑、平衡、宽松三套候选大纲供用户切换。

变化三，AI 文案改写接入。引擎不再只产出结构化大纲（v1.0），还需对每块内容生成三级口吻（孩子向、母亲向、父亲向）的人读摘要。

变化四，数据底座扩展。从 v1.0 的城市、景区、玩法、模板四张表扩展为十一张表，新增餐厅、酒店、交通、医院、商场、用户评价、用户偏好画像。

变化五，工程归属明确。拼装引擎 + AI 抽象层 + 数据底座统一落在 apps/travel-guide（Next.js + Prisma + TipTap），不再散落在 main-site（Vue3 前端）。

第三节 数据模型
================================================================

一、模型总览

走天下 v1.1 共十一张核心表，按用途分四组：

基础地理组：城市 City
内容资产组：景区 Spot、餐厅 Restaurant、酒店 Hotel、交通 Transport、商场 Mall、医院 Hospital
模板与生成组：行程模板 ItineraryTemplate、用户偏好画像 UserTravelPreference、攻略草稿 GuideDraft
评价与结果组：景点评价 SpotReview、旅行明细 TravelRecord（与勋章联动）

二、模型关系图

一个 City 包含多个 Spot、Restaurant、Hotel、Transport、Mall、Hospital。
一个 Spot 可以接收多条 SpotReview。
一个 ItineraryTemplate 关联一个 City，并引用若干 Spot、Restaurant、Hotel。
一个 User 可以有多份 UserTravelPreference、一份 GuideDraft 集合、多条 TravelRecord。
一份 TravelRecord 关联一份 City、一个 ItineraryTemplate（可选）、一份 TravelBadge 集合。

三、关键字段摘要

City：name、coverImage、kidHook、momHook、dadHook、bestSeasons、tags。
Spot：name、cityId、kidHighlights 数组、tips 数组、避坑 pitfalls 数组、recommendedMonths 数组、durationMinutes、kidScore、momScore、dadScore、images。
Restaurant：name、cityId、hasHighChair、isKidTolerant、cuisine、avgPricePerPerson、tags。
Hotel：name、cityId、hasFamilyRoom、hasKidsPool、hasKidsBreakfast、avgPricePerNight、tags。
Transport：fromCityId、toCityId、mode、durationMinutes、priceReference、notes。
ItineraryTemplate：name、cityId、days、season、tags、skeleton（每天的主题与时间块结构）。
UserTravelPreference：userId、homeCity、childAges 数组、travelStyle、budgetLevel、preferredSpots 数组。
GuideDraft：userId、params JSON、outline JSON、contentHtml、status（draft / outline / generated）、mode（chat / quick）。
SpotReview：spotId、userId、rating、text、childAges、visitDate。
TravelRecord：userId、cityId、startDate、endDate、costCents、companions 数组、guideDraftId。
TravelBadge / TravelBadgeDef：勋章发放与定义，沿用 v1.0。

四、用户偏好画像写入时机

UserTravelPreference 由三种事件驱动更新：
其一，用户完成一次向导生成后，把 TravelParams 中稳定的字段（homeCity、childAges、travelStyle、budgetLevel）回写到该表。
其二，用户在个人中心手动补充家庭成员信息。
其三，用户调用懒人一键模式但反问字段的最终选择被记录。

五、SpotReview 反作弊

SpotReview 接入现有审核服务（apps/moderation）。规则如下：
其一，单用户对同一 Spot 一周内最多写一条评价。
其二，文字长度 30–500 字，超出范围拒收。
其三，敏感词过滤（复用 DFA）。
其四，疑似水军（同一 IP 大量提交）触发人工复审。
其五，新用户（前 3 条评价）默认进入"待审"状态。

详细字段定义见附录 A。

七、对话引导模式页面结构

第一屏：聊天流。
顶部固定用户头像与"再开一段新旅程"按钮。

第五节 拼装引擎 + AI 对话式向导双引擎算法
================================================================

一、双引擎角色分工

走天下 v1.1 的核心是两套引擎协同工作：

引擎 A：拼装引擎（Assembler）。
角色：把 TravelParams 转化为结构化行程大纲。
特性：纯规则、无网络依赖、确定性输出（v1.0 拼装引擎升级版）。
输入：TravelParams（出发地、目的地、天数、人数、儿童年龄、季节、偏好、预算、住宿）。
输出：结构化行程大纲对象（含三套候选：紧凑 / 平衡 / 宽松）。

引擎 B：AI 对话引擎（ChatWizard）。
角色：把用户自然语言对话转化为 TravelParams（对话模式），或对结构化大纲进行三级口吻改写（生成模式）。
特性：基于 LLM 的多轮对话与流式输出，必须受数据底座强约束。
输入：用户对话历史 / 结构化大纲 + 口吻档。
输出：TravelParams JSON / 三级自然语言摘要。

两套引擎的边界：
其一，引擎 A 绝不调用任何外部 API，无网络依赖。
其二，引擎 B 绝不允许编造数据，所有内容必须基于已收录的 City、Spot、Restaurant、Hotel、Transport、pitfalls。
其三，所有用户最终看到的"攻略全文"必须经过引擎 A 输出大纲 + 引擎 B 改写两步，不允许任何一条绕过。

二、引擎 A 拼装引擎（升级版）

引擎 A 沿用 v1.0 的五步流水线，但新增三套候选输出。

第一步，模板选择。基于目的地、季节、天数，从 ItineraryTemplate 表匹配最佳模板。匹配策略：目的地优先级最高，其次按天数差异最小，最后按季节匹配。

第二步，景点筛选。从模板骨架出发，按用户儿童年龄数组取交集筛选 Spot，要求每个 Spot 的 recommendedMonths 覆盖用户儿童年龄。

第三步，填补与替换。模板骨架中的每个 Spot 替换为已筛选的具体 Spot。若筛选结果不足，按 kidScore 降序补足。

第四步，节奏调整。根据 Spot 推荐游玩时长，校验每天时间块是否过满。过满则降低下午某 Spot 时长或舍弃某 Spot。午休与晚间哄睡窗口强制保留。

第五步，输出对象构造。补齐每块备注、生成总览文本，输出大纲。

新增逻辑：根据 travelStyle 输出三套候选。

紧凑候选：每天安排 2 个 Spot + 1 个 Restaurant，无自由时间块，午休缩短为 1 小时。
平衡候选：每天安排 2 个 Spot + 1 个 Restaurant + 1 个自由块，午休 2 小时。
宽松候选：每天安排 1 个 Spot + 1 个 Restaurant + 2 个自由块，午休 2.5 小时。

每套候选独立运行上述五步流水线，输出独立大纲对象。

三、引擎 B AI 对话引擎（新增）

引擎 B 由三部分组成：对话参数提取、口吻改写、智能补全。

（一）对话参数提取

输入：用户消息历史（多轮）+ 当前累积 TravelParams。
输出：更新后的 TravelParams + 是否反问的判定。

实现要点：
其一，AI 每轮回复末尾强制 JSON 块输出累积字段，格式严格固定。
其二，系统解析 JSON 块更新 TravelParams，未提及字段保持原值。
其三，AI 同时判断是否还需要更多信息：必填字段未填则继续提问，偏好字段缺失则尝试用默认值填充。
其四，AI 必须在对话中显式说出"目前我了解到的是……是否还有补充？"，便于用户检查累积状态。

字段提取规则（按重要性排序）：
第一优先，必填字段：destination、days、childAges、startDate 附近季节。
第二优先，重要偏好：spotTypes、transport。
第三优先，辅助偏好：budget、hotelReq、travelStyle（紧凑 / 平衡 / 宽松）。

（二）口吻改写

输入：结构化大纲 + 口吻档（kid / mom / dad）。
输出：每块自然语言摘要，Markdown 格式。

口吻档差异：

kid 口吻：第二人称"你"，短句，多感叹号，配 emoji（✅ 🚗 😋 等），推荐"这个超级好玩！"。
mom 口吻：第一人称叙述，柔和语调，强调舒适度与安心细节，配"妈妈可以放心……"、"记得带……"。
dad 口钩：第一人称叙述，简洁直接，强调行动与效率，配"先……然后……"、"关键看……"。

实现要点：
其一，口吻档是 system prompt 的角色变量，不影响内容事实。
其二，每块生成前注入该 Spot/Restaurant/Hotel 的 kidHighlights / pitfalls / tips 原文，AI 不允许修改事实。
其三，禁止使用 emoji 替代图标（沿用 v1.0 UI 规则，AI 输出不含 emoji 替代图标）。

（三）智能补全（懒人模式专属）

输入：用户填写的必填字段（5 个核心）+ 用户历史偏好（UserTravelPreference）+ 目的地画像（City 表）。
输出：补全后的 TravelParams + 每项置信度。

三档置信度阈值（决策 5）：

高置信度（≥0.7）：直接补全，以"我帮你填了 XX"形式列出，无操作按钮。例如：根据 UserTravelPreference.childAges 直接推断预算。
中置信度（0.4–0.7）：补全但提示，以"我默认帮你选 XX，可以改成其他"形式列出。例如：根据目的地最佳季节推断出行月份。
低置信度（<0.4）：必反问，弹窗形式引导用户选择。例如：孩子的兴趣偏好。

反问规则：
其一，单次反问最多 2 个字段，避免疲劳。
其二，反问以选择题形式呈现，每个问题最多 4 个选项。
其三，反问必须解释"为什么问这个"，便于用户理解。

四、AI Provider 抽象层

为支持多供应商切换（决策 1），apps/travel-guide 新增 src/lib/ai/ 目录，统一接口契约：

```
interface AiProvider {
  chat(messages: ChatMessage[], opts?: ChatOptions): AsyncIterable<string>
  json<T>(messages: ChatMessage[], schema: ZodSchema<T>, opts?: ChatOptions): Promise<T>
}
```

首批实现两个 provider：
其一，SiliconFlowProvider：默认 provider，使用 DeepSeek-V2 模型。
其二，OpenAICompatibleProvider：通用 OpenAI 兼容协议 provider，可承载 Qwen / 智谱 / 月之暗面等。

切换机制：
其一，环境变量 AI_PROVIDER=siliconflow | openai_compatible 指定默认。
其二，请求级可指定 provider，便于 A/B 测试。

详细接口契约见附录 E。

五、Prompt 数据约束模板

所有引擎 B 调用都必须注入数据底座约束，禁止 AI 凭空生成：

```
你只能使用以下清单中的景点、餐厅、酒店，不得编造：

{city.name} 已收录景点（{n} 个）：
{spots_json}

{city.name} 已收录餐厅（{n} 个）：
{restaurants_json}

{city.name} 已收录酒店（{n} 个）：
{hotels_json}

{city.name} 已收录避坑（{n} 条）：
{pitfalls_json}

如果用户问到的内容不在清单中，请明确告知"我目前还没有收录这个信息"，并引导用户进入收藏 / 补充流程。
```

约束强度：
其一，AI 在回复中提到的每个景点 / 餐厅 / 酒店必须能在清单中找到 ID。
其二，后端在收到 AI 回复后做一次校验，未匹配的实体强制替换为"待收录"占位。

六、引擎 A 与引擎 B 的执行顺序

对话引导模式流程：
其一，引擎 B 提取参数（多轮）→ TravelParams。
其二，引擎 A 拼装三套候选大纲 → 候选列表。
其三，用户确认某套大纲。
其四，引擎 B 口吻改写（流式输出）→ 完整攻略。
其五，用户保存为草稿或直接发布。

懒人一键模式流程：
其一，用户填 5 个必填字段。
其二，引擎 B 智能补全 + 反问 → TravelParams。
其三，引擎 A 拼装三套候选大纲 → 候选列表。
其四，用户确认某套大纲。
其五，引擎 B 口吻改写（流式输出）→ 完整攻略。
其六，用户保存为草稿或直接发布。

七、引擎性能与成本预估

引擎 A：v1.1 数据规模（十大目的地、五十至一百景点、二十至三十家餐厅），单次运行 < 50 毫秒。三套候选并行运行 < 100 毫秒。

引擎 B 对话提取：单轮 2–4 秒，每段对话平均 4–6 轮。
引擎 B 口吻改写：按大纲长度 5–15 秒，流式输出，用户感知无等待。
引擎 B 智能补全：单次 1–2 秒。

成本预估：DeepSeek-V2 单次对话成本约 ¥0.01–0.03，懒人模式单次补全 ¥0.005，单次完整攻略生成 ¥0.05–0.10。首年预计月均 5 万次生成，月成本约 ¥3000–5000。

八、不做的 AI 行为

明确以下行为不在 v1.1 引擎 B 能力范围内：

不基于历史偏好做目的地推荐（v1.1 只把偏好作为输入参数，不做反向推理）。
不解释攻略内容（攻略详情页文案均为策划手工撰写，不调用 AI 改写）。
不与客服对话交互（无客服对话能力）。
不做攻略评论 / 笔记生成（UGC 内容留给 v2 评估）。

这些行为若未来需要，需立项专项开发。

第十二节 风险与决策记录
================================================================

一、风险与缓解

风险一：AI 供应商价格波动或服务不稳定。
缓解：抽象层已支持多供应商切换，单一供应商故障 1 小时内可切换。

风险二：AI 编造数据导致攻略内容不真实。
缓解：Prompt 数据约束模板 + 后端实体校验 + 强制"清单外占位"。

风险三：对话模式收敛慢，用户体验差。
缓解：每轮强制 JSON 输出累积字段；UI 抽屉展示累积状态；超过 8 轮未收敛主动提示用户改用懒人模式。

风险四：懒人模式反问过多导致用户流失。
缓解：单次反问最多 2 个字段；三档置信度阈值；高置信度字段免打扰。

风险五：数据底座收录不足，AI 回复反复说"未收录"。
缓解：v1.1 数据底座与拼装引擎同步上线，AI 只能在已收录范围内回复。

风险六：跨域调用增加部署复杂度。
缓解：travel-guide 已是独立部署，仅需 Nginx 加一条 location 与 CORS 配置。

二、关键决策记录（v1.1 新增）

决策一：AI 供应商架构采用多供应商抽象层，理由是供应商锁定风险大、便于 A/B 测试。
决策二：AI 拼装引擎 + 数据底座落在 apps/travel-guide，理由是已有 Prisma + TipTap 同栈，与 AI 攻略向导原方案对齐。
决策三：AI 文案改写采用三级口吻（孩子向 / 母亲向 / 父亲向），理由是与 v1.0 三级钩子（kidHook / momHook / dadHook）呼应，覆盖不同阅读场景。
决策四：对话 UI 形态采用 mobile 原生 Vue3，不走 H5 嵌入，理由是避免 H5 与原生壳子之间的样式断层，SSE 流式 uni-app H5 原生支持。
决策五：懒人模式反问阈值采用三档（0.7 / 0.4 / 反问），理由是高置信度免打扰、中置信度透明化、低置信度必反问。

三、待澄清或未来评估

项一：未来是否支持 AI 自动匹配攻略封面图（Unsplash API），AI 攻略向导原 v1.1 增强项。
项二：未来是否支持英文版攻略生成。
项三：未来是否支持多套候选大纲之外的"自定义调整"模式（用户手动改骨架）。
项四：未来是否把 AI 对话能力开放给客服场景（旅行保险、紧急救援等）。

附录 C 接口扩展定义（主站 → travel-guide 跨域）
================================================================

本附录列出 apps/main-site 调用 apps/travel-guide 的三个接口定义。

通用约定：
其一，baseURL：https://travel.grandand.com
其二，认证：请求头 Authorization: Bearer <grandkidsgo_token>
其三，CORS：travel-guide 允许 grandand.com、travel.grandand.com 两个 origin
其四，返回格式：application/json，错误格式 { error: { code, message } }

接口一：GET /api/user/travel-stats
请求：无 body
返回：
{
  totalTrips: 整数,
  totalDays: 整数,
  savedGuides: 整数,
  activeGuides: 整数,
  lastTripDate: ISO 日期或 null,
  topCities: [{ cityId, cityName, count }, ...]（最多 3 个）
}

接口二：GET /api/user/travel-records
请求 query：page（默认 1）、pageSize（默认 20）、startDate、endDate
返回：
{
  total: 整数,
  page: 整数,
  pageSize: 整数,
  records: [{
    id, cityId, cityName, startDate, endDate, costCents, companions, badges: [{ id, name, iconUrl }]
  }, ...]
}

接口三：GET /api/user/travel-badges
请求：无 body
返回：
{
  total: 整数,
  nextBadge: { id, name, condition, progress } 或 null,
  badges: [{
    id, name, iconUrl, description, category, obtainedAt, isRepeatable: false
  }, ...]
}

接口四（新增）：POST /api/wizard/chat（SSE 流式）
请求 body：
{
  messages: [{ role: 'user' | 'assistant' | 'system', content }, ...],
  currentParams: TravelParams JSON,
  userId: 字符串
}
响应：Content-Type: text/event-stream
每行 NDJSON：
{ type: 'message', content: '...' }
{ type: 'params', params: TravelParams }
{ type: 'done' }
错误：{ type: 'error', code, message }

接口五（新增）：POST /api/wizard/assemble
请求 body：TravelParams JSON
返回：
{
  candidates: [{
    style: 'compact' | 'balanced' | 'relaxed',
    outline: { days: [{ day, theme, timeBlocks: [...] }] }
  }, ...]
}

接口六（新增）：POST /api/wizard/rewrite
请求 body：
{
  outline: 结构化大纲 JSON,
  tone: 'kid' | 'mom' | 'dad'
}
返回：Content-Type: text/event-stream，流式输出 Markdown 文本

附录 E AI Provider 抽象层接口契约
================================================================

本附录给出 apps/travel-guide/src/lib/ai/ 的 TypeScript 接口契约与目录结构。

一、目录结构

src/lib/ai/
├── types.ts（公共类型）
├── provider.ts（AiProvider 接口定义）
├── registry.ts（Provider 注册表）
├── index.ts（顶层导出）
├── siliconflow.ts（SiliconFlowProvider 实现）
└── openai-compatible.ts（OpenAICompatibleProvider 实现）

二、公共类型

```typescript
export interface ChatMessage {
  role: 'system' | 'user' | 'assistant'
  content: string
}

export interface ChatOptions {
  model?: string
  temperature?: number
  maxTokens?: number
  signal?: AbortSignal
}

export interface JsonOptions extends ChatOptions {
  schema: import('zod').ZodSchema<any>
}
```

三、Provider 接口

```typescript
export interface AiProvider {
  readonly name: string

  /**
   * 流式对话，逐 chunk 返回生成内容
   */
  chat(
    messages: ChatMessage[],
    opts?: ChatOptions
  ): AsyncIterable<string>

  /**
   * 结构化 JSON 输出，使用 zod schema 校验
   */
  json<T>(
    messages: ChatMessage[],
    opts: JsonOptions
  ): Promise<T>
}
```

四、注册表与切换

```typescript
export interface AiRegistry {
  get(name: string): AiProvider
  default(): AiProvider
  register(name: string, provider: AiProvider): void
}
```

环境变量：
AI_PROVIDER=siliconflow | openai_compatible（默认 provider 名称）
SILICONFLOW_API_KEY、ENDPOINT、MODEL
OPENAI_COMPATIBLE_API_KEY、ENDPOINT、MODEL
