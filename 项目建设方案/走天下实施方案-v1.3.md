零、用户价值宣言（v1.3 新增，本节为整份文档的纲领）

走天下要回答的根本问题是：为什么我们要用 AI 拼装引擎给用户生成出行计划，而不是让用户自己拼？

答案分两层：孩子视角决定产品的内容标准，父母视角决定产品的服务对象。

第一层，孩子视角——决定产品的内容标准（孩子是产品的主角，不是叙述者）。

其一，孩子是产品的主角。亲子出行的所有产品（计划、评价、攻略、景点卡片、餐厅推荐），孩子都是主角，不是父母。计划的每个时间块、每张景点卡片、每条餐厅推荐，都要回答"这趟行程里孩子在做什么、状态如何"。攻略最有价值的部分，是"我家 3 岁娃在这家博物馆只待了 20 分钟就哭了，但隔壁科技馆他玩了 2 小时没闹"——这种颗粒度，机器拼不出来，只有真实带孩子去过的人才知道。

其二，孩子不是叙述者。这一点必须澄清——"孩子视角"≠"孩子口吻"。计划、攻略、评价的叙述者始终是妈妈或爸爸，口吻是大人的。一篇攻略的典型表述是「Day 1 上午 X 馆：孩子在这里玩疯了，跳来跳去不肯走」，不是「我（孩子）在这里玩得好开心」。主体是孩子（关注孩子的体验），叙述者是妈妈（用大人的口吻描述孩子）。这两个不能混。

其三，孩子是关注点本身。孩子的具体年龄、体力曲线、注意力时长、兴趣点，是产品内容的核心要素，不是辅助标签。3 岁娃和 8 岁娃面对同一个景点的停留时长、餐厅要求、是否需要午休，是截然不同的——如果计划的每个时间块不是围绕孩子的状态排的，行程一上来就废了。

进一步说，孩子大部分要么很小，要么去过的地方很少，要么根本不知道选哪个，所以走天下的工作不是"提供素材让孩子/父母自己选"，而是：

步骤一，主动收集孩子画像。出行前必须先建画像（年龄、性别、平时喜好、过敏忌口、体力情况、是否晕车、是否怕生等）。这些信息由平台主动询问，是出行流程的必经步骤，不是可选优化项。

步骤二，在多库里联动筛选。不是只看景点库，而是城市库（适不适合这个年龄段的孩子）、景点库（是否对年龄段友好）、酒店库（是否有亲子房/儿童早餐）、公园库（是否有儿童设施）、游乐场库（年龄上限是否合适）、餐厅库（是否有儿童餐椅/儿童菜单）——必须联动筛选，因为"孩子住得舒服但景点玩得累"或"景点好玩但孩子住不下"都是失败。

步骤三，多要素综合排序。同样适合 3 岁娃的 5 个景点，最终选哪 2 个进入当天的行程，要综合评价（其他家长打分）、行程（顺路不顺路）、成本（门票+交通）、时间（开放时间+孩子合适时段）、出片率（家长能不能拍出好看的照片）——不是单要素，是多要素加权。

第二层，父母视角——决定产品的服务对象（方案要合理，要父母认可）。

其一，父母是决策者，平台不替父母做主。走天下的所有方案，最终都要父母认可后才能生效。父母有权否决、有权微调、有权追加约束——这不是"按确认键"的交互，是"决策伙伴"的交互。一键生成的方案，必须附带"为什么这么排"的理由，让父母能判断合理性；父母否决时，可以输入否决理由，引擎根据理由重新排。

其二，父母的拼接成本极高。一位妈妈带孩子出行的执行成本有三层：

执行成本一，掌握的信息太少。她要在小红书、点评、马蜂窝、地图、天气预报之间反复比对，哪些景点对孩子友好、哪些餐厅有儿童餐椅、景点之间距离多远、几点到几点玩合适、车停哪、孩子会不会累趴。

执行成本二，大量信息要做拼接。即便她掌握了所有素材，「拼接」本身也是巨大成本——A 景点玩完去 B 餐厅吃饭还是回酒店休息，下午能不能撑住第二个景点，要不要把 C 博物馆挪到第二天。拼接错误的代价，往往是「孩子现场崩溃」或「父母全程焦虑」。

执行成本三，多段行程（连程）尤其难。A 城市→B 城市→C 城市这种 5 天以上的行程，住宿要不要跟着换城市、动线是单向还是回头、不同城市之间的交通是动车还是飞机、行李怎么寄存，没有现成模板。

这三件执行成本，是父母自己拼不出来的。市场上大多数亲子旅行内容平台只解决执行成本一（提供素材），不解决执行成本二（自动拼接）、执行成本三（连程衔接）。童慧行走天下的差异化就在这里——我们用 AI 拼装引擎把执行成本二和执行成本三自动化。

总结成一句话：孩子视角决定产品的内容标准（孩子是主角，叙述者是妈妈），父母视角决定产品的服务对象（方案要合理，要父母认可）。

为了兑现这个区分，我们用 AI 拼装引擎做出以下五条产品价值承诺：

承诺一，孩子画像是合理方案的必经输入。父母出行前必须先建立孩子的画像（年龄、性别、平时喜好、过敏忌口、体力情况等），平台基于画像从多库（城市/景点/酒店/公园/游乐场/餐厅）联动筛选，结合多要素（评价、行程、成本、时间、出片率）综合排序，生成合理方案。这是孩子视角落到产品里的具体形态——不是"平台替孩子选"，而是"平台基于孩子画像给父母合理方案"。

承诺二，自动计算。用户填完出发地、目的地、孩子画像、出行天数、住宿档次，引擎在 100 毫秒内输出三套候选行程大纲，不让父母自己去算。这是兑现"执行成本二"的工程承诺。

承诺三，不绕路、不走回头路。每天点位按地理就近排序，多天行程按地理单向推进，A→B→C 之间不重复折返；父母不需要懂地理，引擎给出当天动线图。这是兑现"执行成本三"中动线部分的核心算法。

承诺四，省时省钱舒服三档可选。三套候选不再只是节奏差异（紧凑/平衡/宽松），而是维度差异——省时档优先压缩交通时长、压缩景点间隔；省钱档优先公共交通、平价酒店；舒服档优先慢节奏、午休、长景点停留。父母根据自家情况选一档。

承诺五，连程自动衔接。A→B→C 多段行程，住宿跟着走、交通按段切分、动线单向不回头，引擎自动计算每一段的最优衔接方式（高铁/飞机/自驾）。

上述五条承诺，是童慧行走天下作为产品的「存在理由」。承诺一是内容标准的承诺（孩子视角落到产品里的具体形态），承诺二至五是工程方法的承诺（替父母降低执行成本）。本方案后续每个章节、每个接口、每个页面都要回答一个问题：「它如何兑现这五条承诺？」如果某项设计无法回答这个问题，它就不该存在。

一、产品定位

童慧行走天下是童慧行生态下的亲子旅行内容平台。产品由三大功能构成：
功能一，计划。出行前，用户通过 AI 向导生成专属出行计划，可手动调整。形态为日程表 + 景点卡片 + 餐厅卡片 + 避坑提醒。
功能二，攻略。出行后，用户基于出行中的现场记录（照片/视频/体会/孩子评分），一键发布为带富媒体的攻略。攻略对所有用户可见、可收藏、可被"一键生成我的版本"。
功能三，逛。任何时候，用户可浏览他人发布的攻略。逛是流量入口与留存引擎。

三大功能相互连接，构成自循环产品。

二、产品主链路（钉死，整份文档以此为纲）

```
首页逛（攻略信息流）
  ├─ 行为 A：随便逛（信息流刷到感兴趣攻略）
  └─ 行为 B：特意搜索（搜索框输入关键词）
       ↓
     命中攻略 → 进入攻略详情页
       ↓
     兴趣触发：收藏 / 点赞 / 关注作者
       ↓
     决策触发：一键生成"我的版本"
       ↓
     用户修改重要信息（出行日期 / 孩子年龄 / 预算 / 替换景点）
       ↓
     生成我的计划（PlanRecord，status=confirmed）
       ↓
     进入出行阶段
       ↓
     出行中：现场记录（照片 / 视频 / 体会 / 孩子评分）
       ↓
     完成出行
       ↓
     一键发布为攻略（Guide，status=pending_review）
       ↓
     审核通过 → published → 被其他用户逛到
       ↓
     回到首页逛（循环启动）
```

整份文档每个章节都要回答："它如何服务这条主链路？"

三、与已有产品关系

走天下与论坛（apps/forum）的关系：
走天下做"亲子旅行攻略"垂直内容；论坛做综合讨论社区。两者用户群有部分重叠（家长群体），但内容形态不同。
v1.2 决定走天下独立运营，与 forum 不联动（决策 14）。评论/问答等深度社区功能留给 forum，走天下的互动限定为点赞/收藏/关注作者（轻互动）。

走天下与商城（apps/store）的关系：
完成出行、获得勋章等行为可兑换商城物品（v2 评估）。

走天下与主站用户中心：
主站个人中心新增"走天下记录"模块，展示旅行统计与勋章。接口来源是 apps/travel-guide。

四、非目标

不做独立应用商店上架。
不做旅游产品交易（导游预订、票务、保险）—v2 评估。
不做跨语言版本（v1.2 仅简体中文）。
不做评论/问答等深度社区功能（留给 forum）。
不做 AI 自动生成攻略正文（所有正文都来自用户的真实记录，AI 只润色）。

第三节 数据模型
================================================================

一、模型总览

走天下 v1.2 共十九张核心表，按用途分七组：

基础地理组：城市 City、景区 Spot、餐厅 Restaurant、酒店 Hotel、交通 Transport、商场 Mall、医院 Hospital
模板与生成组：行程模板 ItineraryTemplate、用户偏好画像 UserTravelPreference
用户产出组：计划 PlanRecord、计划媒体 PlanMedia、计划笔记 PlanNote、孩子评分 ChildRating
公共内容组：攻略草稿 GuideDraft、攻略 Guide、攻略收藏 GuideSave、攻略点赞 GuideLike、用户关注 UserFollow
评价与结果组：景点评价 SpotReview、旅行明细 TravelRecord
勋章组：勋章定义 TravelBadgeDef、勋章发放 TravelBadge

二、模型关系图

一个 City 包含多个 Spot、Restaurant、Hotel、Transport、Mall、Hospital。
一个 Spot 可以接收多条 SpotReview。
一个 ItineraryTemplate 关联一个 City，并引用若干 Spot、Restaurant、Hotel。
一个 User 可以有多份 UserTravelPreference、多份 GuideDraft、多份 PlanRecord、多条 TravelRecord、多份 Guide。
一份 PlanRecord 关联一个 City（可选）、一份 sourceGuide（可选）、一个 ItineraryTemplate（可选）。
一份 PlanRecord 包含多份 PlanMedia、多条 PlanNote、多条 ChildRating。
一份 PlanRecord 可发布为一份 Guide（通过 sourcePlanRecordId 关联）。
一份 Guide 可被多份 GuideSave/GuideLike 引用。
一个 User 可关注多个其他 User（通过 UserFollow）。

三、关键字段摘要

（一）基础地理组

City：name、coverImage、kidHook、momHook、dadHook、bestSeasons、tags。
Spot：name、cityId、kidHighlights、tips、pitfalls、recommendedMonths、durationMinutes、kidScore、momScore、dadScore、images、tags。
Restaurant：name、cityId、hasHighChair、isKidTolerant、cuisine、avgPricePerPerson、tags。
Hotel：name、cityId、hasFamilyRoom、hasKidsPool、hasKidsBreakfast、avgPricePerNight、tags。
Transport：fromCityId、toCityId、mode、durationMinutes、priceReference、notes。
Mall：name、cityId、hasKidsPlayArea、tags。
Hospital：name、cityId、hasPediatrics、address、phone。

（二）模板与生成组

ItineraryTemplate：name、cityId、days、season、tags、skeleton（JSON）。
UserTravelPreference：userId、homeCity、childAges、travelStyle、budgetLevel、preferredSpots。

（三）用户产出组（v1.2 新增）

PlanRecord：userId、cityId、sourceGuideId（可选，从哪份攻略生成）、startDate、endDate、travelers（成人/儿童人数）、childAges、status（draft / confirmed / active / completed / published）、timelineBlocks（JSON，实际行程结构）、createdAt、updatedAt。

PlanMedia：planRecordId、type（photo / video）、ossKey（阿里云 OSS key）、takenAt、spotId（可选，关联到哪个景点）、caption、orderIndex。
（用户通过阿里云 OSS Web 端直传上传，不经 travel-guide 中转，详见附录 F。）

PlanNote：planRecordId、spotId（可选）、text、mood（happy / tired / excited / challenging）、createdAt。

ChildRating：planRecordId、spotId、rating（1–5）、emoji（👍 / 😄 / 😐 / 😣 / 🤢）、recordedAt。

（四）公共内容组

GuideDraft：userId、params、outline、contentHtml、status（draft / outline / generated）、mode（quick）。

Guide：userId、sourcePlanRecordId（从哪份计划发布）、title、coverImages（OSS key 数组）、contentHtml（富文本含图片/视频）、cityId、days、childAges、travelStyle、season、status（pending_review / published / rejected）、publishedAt、viewCount、saveCount、likeCount、tags、createdAt、updatedAt。

GuideSave：userId、guideId、createdAt。（唯一索引 userId+guideId）

GuideLike：userId、guideId、createdAt。（唯一索引 userId+guideId）

UserFollow：followerId、followeeId、createdAt。（唯一索引 followerId+followeeId）

（五）评价与结果组

SpotReview：spotId、userId、rating、text、childAges、visitDate、status（pending / approved / rejected）。
TravelRecord：userId、cityId、startDate、endDate、costCents、companions、planRecordId。

（六）勋章组

TravelBadgeDef：勋章定义，含条件结构化字段。
TravelBadge：userId、badgeDefId、obtainedAt、planRecordId（触发该勋章的计划）。

四、索引建议

PlanRecord：userId + status 复合索引、startDate 索引。
PlanMedia：planRecordId + orderIndex 复合索引。
Guide：status + publishedAt 复合索引（feed 查询）、cityId 索引、tags 数组索引。
GuideSave：userId + createdAt 复合索引（"我的收藏"列表）。
GuideLike：userId + createdAt 复合索引。
UserFollow：followerId + followeeId 复合索引。

六、SpotReview 反作弊（沿用 v1.1，本节不重复）

================================================================

一、整体架构

走天下 v1.2 涉及三个 app 的协同：

apps/travel-guide（Next.js 14 + Prisma）：承载全部后端能力。
其一，拼装引擎 + AI 抽象层（src/lib/ai/、src/lib/assembler/）。
其二，数据底座 Prisma 表（19 张）。
其三，攻略 / 计划 / 互动 / 勋章 API 集合。
其四，OSS STS 直传凭证签发。
其五，moderation 审核对接。

apps/mobile（uni-app Vue3）：承载走天下 tab 全部页面。

第五节 拼装引擎与 AI 能力边界

一、双引擎角色分工（v1.3 修订）

走天下 v1.3 的核心仍是两套引擎协同工作。所有设计都服务第一节价值宣言——孩子视角决定产品的内容标准（孩子是主角，叙述者是妈妈），父母视角决定产品的服务对象（方案要合理，要父母认可）。

引擎 A：拼装引擎（Assembler）。
角色：基于孩子画像 + 多库联动 + 多要素加权，给父母算出一份合理行程大纲。
特性：纯规则、无网络依赖、确定性输出。
输入：TravelParams（含孩子画像 ChildProfile）+ 可选 sourceGuideId。
输出：结构化行程大纲对象（三档候选：省时 / 省钱 / 舒服，每档内部再分紧凑 / 平衡 / 宽松节奏）。

引擎 B：AI 引擎（AiEngine）。
角色：对结构化大纲进行口吻改写（可选润色）、解答用户对攻略的提问、对行程合理性做智能问答。
特性：基于 LLM，受数据底座强约束，不凭空生成数据。
输入：结构化大纲 + 口吻档 / 攻略内容 + 用户问题。
输出：自然语言改写 / 攻略问答答复 / 行程合理性问答。

边界：
其一，引擎 A 绝不调用任何外部 API。
其二，引擎 B 绝不允许编造数据，所有内容必须基于已收录的 City、Spot、Restaurant、Hotel、Transport、pitfalls、ChildProfile。
其三，所有用户最终看到的"攻略全文"必须基于用户的真实记录（PlanMedia / PlanNote / ChildRating），AI 只润色、不生成主体。
其四，引擎 B 绝不替父母做主。AI 给出的行程建议、问答答案都只是参考，最终是否采纳由父母决定（详见承诺一）。

二、引擎 A 拼装引擎（v1.3 升级版）

（一）支持 sourceGuideId

拼装引擎 v1.3 接受可选的 sourceGuideId 参数：
有 sourceGuideId：保留源攻略的景点/餐厅清单，按用户新参数（孩子画像、出行日期、预算）微调。
无 sourceGuideId：从零拼装（三档候选）。

（二）核心输入：孩子画像 ChildProfile

拼装引擎的输入 TravelParams 必须包含至少一份 ChildProfile（一个家庭可有多份），每份含以下字段：

其一，基本信息。年龄（精确到月龄）、性别、当前身高体重（用于判断是否需要安全座椅、是否可玩某些身高限制项目）。

其二，平时喜好。喜欢的题材（动物 / 车辆 / 恐龙 / 海洋 / 公主 / 太空 / 绘本 / 音乐 / 运动等，多选）、喜欢的活动类型（户外 / 室内 / 表演 / 互动 / 静态参观等，多选）、是否有特别讨厌的事物（黑暗 / 高处 / 噪音 / 陌生动物等）。

其三，体力与作息。每天可承受的连续活动时长（小时）、是否需要午休（必填，是 / 否 / 可选）、早睡晚睡（影响每天第一个景点的开始时间）。

其四，限制与禁忌。是否晕车（影响连程交通方式）、过敏食物（影响餐厅筛选）、是否怕生（影响是否安排人少的时段）、是否特殊体质（影响活动强度上限）。

孩子画像由平台主动询问，**是出行流程的必经步骤，不是可选优化项**。父母可在"个人中心"维护一份默认画像，下次出行复用并按需调整。

（三）多库联动筛选（v1.3 新增）

拼装引擎不是单库筛选景点，而是六个库联动筛选：

库一，城市库（City）。基于孩子年龄筛适合的城市——3 岁娃优先城市内有大型动物园 / 海洋馆 / 室内乐园的城市；8 岁娃可加博物馆 / 名胜古迹多的城市。

库二，景点库（Spot）。基于孩子年龄 + 喜好 + 体力筛具体景点——年龄不符的剔除（如身高 1.2m 以下项目对 3 岁娃不合适）、喜好不符的降权、时长过长的剔除。

库三，酒店库（Hotel）。基于孩子年龄 + 同等景点区域筛酒店——必须有亲子房或加床选项、最好有儿童早餐、离当天景点步行或短驾可达。

库四，公园库（Park）。作为景点库的补充，3–6 岁娃每天必须有 1 个户外放风时间块（哪怕 30 分钟）。

库五，游乐场库（Playground）。年龄上限必须合适，且和当天的其他景点动线顺路。

库六，餐厅库（Restaurant）。必须有儿童餐椅、最好有儿童菜单、最好有不辣选项（针对 3 岁以下）。

联动规则：景点确定后，酒店位置必须围绕景点而不是反过来；餐厅位置必须围绕当天的景点动线；游乐场和公园作为景点之间的过渡。

（四）多要素综合排序（v1.3 新增）

同样适合孩子画像的 5 个景点，最终选哪 2–3 个进入当天行程，按以下要素加权：

要素一，评价（其他家长打分 + ChildRating 聚合）。权重 25%。

要素二，行程（顺路程度）。基于当天其他点位，计算动线单向不回头贡献。权重 20%。

要素三，成本（门票 + 交通 + 周边餐饮）。权重 15%。

要素四，时间（开放时间 + 孩子合适时段）。比如 3 岁娃下午 2 点必须午休，那么下午 2 点后的景点必须剔除或换上午。权重 25%。

要素五，出片率（家长能不能拍出好看的照片）。这是 v1.3 新增维度——走天下的用户主要是妈妈，妈妈非常在意"能不能拍出朋友圈好看的照片"，这直接影响她是否会选这个景点。权重 15%。

五要素权重总和 100%，可在运营后台调整，但默认走 25/20/15/25/15 配比。

（五）三档候选：省时 / 省钱 / 舒服（v1.3 修订）

v1.2 的三套候选是"紧凑 / 平衡 / 宽松"，本质是节奏差异。v1.3 升级为"省时 / 省钱 / 舒服"三档，**本质是维度差异**——每档内部再分紧凑 / 平衡 / 宽松节奏。

省时档：优先压缩交通时长、压缩景点间隔。每天景点排满但节奏紧凑，适合假期短的出行。

省钱档：优先公共交通（地铁 / 公交 / 步行可达景点）、平价酒店（无星级要求但要求亲子设施齐全）、免费或低价景点。

舒服档：优先慢节奏、午休、长景点停留。每天景点少 1–2 个但每个停留时间更长，午休块不可被用户取消。

每档内部再按紧凑 / 平衡 / 宽松三档细分节奏，合计 9 种组合（实际下不会全用，按用户偏好默认 3 档）。

（六）连程衔接规则（v1.3 新增）

针对 A→B→C 多段行程（5 天以上），引擎必须遵循以下衔接规则：

规则一，住宿跟着走。每个城市的住宿必须选该城市最后一天景点附近，方便最后一天不用赶路直接退房去车站。

规则二，交通按段切分。城市间交通（高铁 / 飞机 / 自驾）必须排在上午或傍晚，不排在中午（中午是孩子的吃饭/午休时间，且赶车容易错过）。

规则三，动线单向不回头。跨城市的整体动线必须单向（比如 上海→杭州→黄山，不要 上海→黄山→杭州→上海）。

规则四，行李动线最短化。带行李出行时，每天的景点必须尽量靠近当天住宿，不做"早上退房、玩一天、晚上到下一家酒店"的拉链式行程——这种行程是孩子崩溃的最大原因之一。

规则五，缓冲日规则。连程超过 5 天的，必须至少预留 1 个"什么都不干"的休息日（或半日），让孩子和父母都缓一缓。

（七）五步流水线（v1.3 升级）

第一步，孩子画像匹配。输入 TravelParams 时，先验证 ChildProfile 完整性，缺字段时提示父母补齐（不允许跳过）。

第二步，模板选择。基于目的地、季节、天数、孩子画像，从 ItineraryTemplate 表匹配最佳模板。

第三步，多库联动筛选 + 多要素排序。模板骨架的每个时间块，按上文（三）（四）的规则填具体点位。

第四步，连程衔接校验。如果是多段行程，按上文（六）的规则校验衔接是否合理，不合理则回退到第三步重排。

第五步，节奏调整 + 三档输出。根据孩子画像里的体力字段，按年龄段下限打 7 折校验每个时间块时长；每天景点上限硬限制；午休块不可被用户取消；输出三档候选（省时 / 省钱 / 舒服）+ 每档的"为什么这么排"理由（用于父母认可环节）。

三、引擎 B AI 引擎

（一）能力清单（v1.3 升级为五类）

AI 在 v1.3 中承担以下五类能力：

其一，攻略润色。用户发布攻略前，可点"AI 润色"，AI 基于用户原文与三级钩子风格（kid/mom/dad）润色文字。润色结果覆盖用户原文需用户手动确认。

其二，攻略问答。攻略详情页右下角悬浮 AI 助手图标，用户点击后可问"这里推车方便吗？""下雨天怎么办？"，AI 基于该攻略的正文与 pitfall 回答。这是面向小红书母婴用户最高频的 AI 用法。

其三，攻略摘要。AI 自动为每篇攻略生成 1 句话摘要（80 字以内），用于 feed 卡片副标题。

其四，封面推荐。从用户上传的多张照片中，挑出"含孩子入镜 + 视觉质量最高"的几张作为封面候选。不做人脸识别（决策 10），仅基于图片特征（人脸占比、构图、清晰度）。

其五，行程智能问答（v1.3 新增）。这是兑现承诺一的关键能力——父母拿到引擎生成的行程大纲后，可以针对行程本身提问，AI 基于拼装引擎输出的结构化大纲 + ChildProfile + 历史行程数据回答。

行程智能问答的具体形态：

问答一，是否绕路。基于当天所有时间块的地理坐标，回答当天是否动线单向、是否有回头路、能否再省 N 分钟交通时间。回答必须引用具体时间块编号。

问答二，孩子能不能撑住。基于 ChildProfile 的体力字段 + 当天所有时间块的累计活动时长，回答 3 岁娃（或指定孩子）能不能撑住、是否建议砍掉哪个时间块、是否需要增加休息。

问答三，连程衔接是否合理。如果是 A→B→C 多段行程，回答每个城市间衔接是否最优（高铁 vs 飞机 vs 自驾）、住宿位置是否合理、行李动线是否最短。

问答四，能否再优化。父母输入优化诉求（"想再省半天"、"想多看一个博物馆"、"想避开周末人多的时段"），AI 基于结构化大纲给出可执行的调整建议。

问答五，否决理由重排。父母对当前方案某一部分不满意，可输入否决理由（"X 馆娃上次去过哭了一小时，换一个"），引擎基于理由重排该时间块并返回新方案。

行程智能问答的关键约束：
其一，回答必须引用具体时间块编号（"Day 2 上午 10:00–11:30 时间块"），不允许给笼统答案。
其二，AI 不替父母做主——所有建议都需要父母手动确认后才能生效。
其三，AI 不凭空生成数据——所有回答必须基于结构化大纲 + ChildProfile + 历史行程数据，不调外部知识。

（二）边界

AI 绝不做的：
绝不编造景点、价格、营业时间。
绝不基于攻略正文之外的内容回答（不调外部知识）。
绝不在攻略正文中插入未由用户上传的内容。
绝不替父母做主——行程智能问答的所有建议都需父母手动确认。
绝不把孩子的口吻当叙述者——AI 润色必须保持妈妈/爸爸的口吻，不能改成孩子口吻。

第十二节 风险与决策记录

一、风险与缓解（v1.2 增补）

风险一：AI 供应商价格波动或服务不稳定。

风险七：首页改为内容首页后，原"AI 生成入口"被弱化。

一、模块定义

逛是用户浏览他人攻略的功能入口，是整个产品的流量入口与留存引擎。

二、入口

走天下 tab 默认首页（内容首页）。

三、首页信息流设计

列表样式：小红书双列瀑布流（图文卡片，孩子入镜图为主）。
标题格式：【目的地 · X 天 X 晚 · X 岁娃】人均 XX 元 + N 个避坑（沿用 v1.0 卡片规范）。
排序算法：热度分 = 收藏数 × 3 + 点赞数 × 2 + 发布时间衰减 + 孩子入镜图分。
子元素：置顶运营位（编辑精选，最多 3 篇）、热门标签横滑。
筛选维度（顶部下拉）：目的地、孩子年龄、出行天数、季节、景点类型。
搜索入口：顶部搜索框，点击进入搜索结果页。

四、两种用户行为收敛

行为 A：随便逛（首页 feed 刷到感兴趣攻略）。
行为 B：特意搜索（搜索框输入关键词）。

两种行为在命中攻略后完全统一——进入详情页 → 收藏/点赞/关注 → 一键生成我的版本。

五、搜索实现

单一搜索框，混合结果排序。
后端一次查询，攻略优先（命中标题/正文/景点名），城市/景区/酒店次之。
实体识别（用户搜的是城市还是景区）通过命名实体识别或简单关键词匹配。
复杂度低，v1.2 上线即可，Elasticsearch 留 v2 评估。

六、详情页设计（攻略详情页）

第十四节 出行情景与记录（出行前-中-后）
================================================================

一、三阶段定义

阶段一：出行前。用户已生成计划（PlanRecord.status=confirmed），等待出发。
阶段二：出行中。用户标记计划进入 active 状态，日期在 startDate 与 endDate 之间。可现场记录。
阶段三：出行后。用户标记计划完成（completed），可发布为攻略。

二、出行前

计划详情页顶部状态条显示"距离出行还有 X 天"。
用户可在此阶段查看计划、做最后调整（替换景点、改日期、调人数）。
用户点"开始出行"，计划 status → active。

三、出行中

计划详情页自动切换为"出行模式"。
顶部状态条显示"出行第 X 天 / 共 X 天"。
每日时间线展开，每块标记按钮显示：
我去了（标记为绿色）。
改去（替换为其他景点，跳转到替换流程）。
没去（标记为灰色，记录原因）。
点击某块展开现场记录面板：
照片上传（系统相机/相册选择后回 app，详见决策 8）。
视频上传（同上）。
文字体会输入。
孩子评分（1-5 星 + emoji 选择）。
避坑标记：用户在某块点"踩坑"按钮，输入一句话心得，自动汇入该景点的 pitfalls（待审核）。

四、出行后

用户标记计划完成（completed）。
顶部出现"发布为攻略"按钮，点击进入发布编辑器。
发布编辑器初始内容由系统自动汇总：
时间线（按实际走过的景点顺序）。
封面候选（AI 推荐的 5 张照片）。
每个景点的"实际照片 + 体会 + 孩子评分"。
用户可手动调整、AI 润色（可选），点"提交发布" → Guide.status = pending_review。

五、本地暂存与补传

网络异常时，PlanMedia/PlanNote/ChildRating 写入本地暂存队列。
网络恢复后自动重试上传。
后端允许在计划 active 与 completed 期间内任何时间点补传，最长 30 天。

第十六节 审核流程（接入 moderation）
================================================================

一、审核策略

攻略文本：发布时自动跑 DFA 敏感词过滤。
DFA 命中 → 直接拒绝发布 + 提示敏感词位置。
DFA 未命中 → Guide.status = pending_review → 进入人工审核队列。

二、人工审核

入口：apps/admin 后台新增"攻略待审"Tab。
审核列表展示：作者头像、攻略封面、标题、正文前 200 字、用户历史发布记录（最近 10 篇的通过率）。
审核操作：通过 / 拒绝（带原因）。
审核结果：通过 → Guide.status = published；拒绝 → Guide.status = rejected + 用户收到通知。

三、审核 SLA

中位时长 ≤ 4 小时。
超时未审核：自动转交下一位审核员。
连续 3 篇超时未审：触发运营告警。

四、申诉

用户被拒绝后可发起申诉，申诉进入复审队列。
复审由高级审核员处理，复审结果为最终结果。

五、SpotReview 审核（沿用 v1.1，本节不重复）

附录 F 阿里云 OSS 直传 STS 接口契约
================================================================

一、设计目标

mobile 端直传照片/视频到阿里云 OSS，不经 travel-guide 中转流量。
travel-guide 仅签发 STS 临时凭证（5 分钟过期）。

二、流程

```
[mobile 端]
  ↓ POST /api/upload/sts { planRecordId, fileType }
[travel-guide 端]
  ├─ 校验 grandkidsgo_token
  ├─ 校验 planRecordId 归属当前用户
  ├─ 调用阿里云 STS 服务签发临时凭证
  └─ 返回 accessKeyId、accessKeySecret、securityToken、expiration、bucket、region、uploadDir
  ↓
[mobile 端]
  ├─ 用临时凭证 PUT 文件到 OSS（uploadDir 路径下）
  ├─ 拿到 OSS key
  └─ POST /api/plans/:id/media { ossKey, ... }
[travel-guide 端]
  └─ 写入 PlanMedia 表
```

三、安全约束

其一，STS 凭证有效期 5 分钟，过期后客户端必须重新申请。
其二，uploadDir 路径前缀绑定 planRecordId，确保用户只能上传到自己的目录下。
其三，OSS bucket 配置防盗链，仅允许 travel.grandand.com 与 grandand.com 两个域名访问。
