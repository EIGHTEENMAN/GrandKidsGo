## 一、3 个核心问题先答

### Q1：用户没去过能打吗？
**当前可以，未来应该限制。** PlaceReview 表对 visitDate 是 `DateTime?`，UI 表单 visitDate 也是可选——所以理论上**没去过也能打分**。

**这有问题**——Tripadvisor / 大众点评都要求必须去过（或上传消费凭证/位置打卡），否则刷分门槛太低。

**建议方案**：
- v1.1 引入「去过凭证」三选一：① 上传照片 ② 打卡定位（手机端有 GPS）③ 选择过去 12 个月的 visitDate（弱凭证）
- 无凭证的评价进 `pending` 队列，标记「未去过」，仅展示在「去过的人评价」下方，独立排序
- v2.0 接 AMAP key 后用 GPS 自动打卡

### Q2：用户在哪里打？
**当前**：地点详情页底部「写评价」表单，弹窗式提交。
**建议扩展 3 个入口**（提高打分密度）：

| 入口 | 触发时机 | 是否要求去过 |
|------|----------|--------------|
| **A. 地点详情页「写评价」按钮**（现有） | 用户专门来打 | 必须（v1.1+） |
| **B. 攻略正文下「你去过吗？来打分」** | 用户看完攻略被激发 | 必须（v1.1+） |
| **C. 个人中心「我的行程」→ PlanRecord 完成页** | 行程结束后系统引导 | 必须（凭证 = PlanRecord） |
| **D. 孩子说 / 儿童画廊发布时「同时评价」** | 已产生 UGC 时顺带打分 | 可选（归为「未去过」） |

**攻略里也能打**——但攻略正文下方应是「去过的人评价」而不是「一键打分」（避免误导读者以为是攻略评分）。攻略本身评分用单独的 `GuideReview` 表。

### Q3：打分维度怎么设置？

| 维度 | 类型 | 必填 | 适用类型 | 权重 |
|------|------|------|----------|------|
| **成人评分**（整体体验） | 1-5 星 | 是 | 全部 | 40% |
| **孩子评分**（孩子真实感受） | 1-5 星 | 是（有孩子） | 全部 | 60%（加权） |
| **年龄适配**（自动计算） | 派生 | 派生 | 全部 | 影响显示 |
| **便利设施**（多选） | bool ×5 | 否 | 全部 | 标签 |
| **kidFriendly 综合** | 1-5 | 否 | 全部 | 备用 |
| **文本评论** | 500 字 | 否 | 全部 | 展示 |
| **标签** | 多选 | 否 | 全部 | 检索 |
| **visitDate** | date | 是（v1.1+） | 全部 | 反作弊 |

**当前已有**：成人 1-5 + 孩子 1-5 + childAgeMonths + 文本 + tags + 5 个 bool 便利 + kidFriendly 综合 + visitDate。

## 二、地点详情页的红蓝灰三个打分

### 现状数据流
```
Place 表字段 kidScore / momScore / dadScore（Float）
   ↓
种子数据写入（seed-place.ts 一开始就写死了 4.5 之类的值）
   ↓
Place 详情页 Hero 标题下方渲染三徽章
```

**这是种子编辑打分，不是用户打分聚合。** Place 表没存用户评价聚合——只有 PlaceReview 表聚合到 Place 上的字段未来要加。

### 建议方案

**v1.0（本次）**：保留种子 `kidScore/momScore/dadScore` 作为「编辑推荐分」兜底，新增 `PlaceScore` 视图（**不是新表，是 PlaceReview 聚合计算**）：

| 显示 | 来源 | 含义 |
|------|------|------|
| 🩷 **孩子 X.X**（粉） | `PlaceReview` 表的 `childRating` 聚合 avg | **真实带孩子去过的家庭评分** |
| 💙 **妈妈 X.X**（蓝/青） | `PlaceReview` 表 `adultRating` 中 `user.gender='female'` 子集 avg | 妈妈视角 |
| 🩶 **爸爸 X.X**（灰） | `PlaceReview` 表 `adultRating` 中 `user.gender='male'` 子集 avg | 爸爸视角 |

**关键变化**：
- 颜色对应「孩子粉 / 妈妈青（蓝青渐变主色）/ 爸爸灰」
- 红蓝灰里**红**不重要——童慧行品牌色是蓝青，**妈妈用青（cyan）**，**爸爸用灰（slate）**
- 数据**完全来自用户评价聚合**，不再是写死的种子
- 评价 < 3 条时**降级显示种子编辑分**，避免空数据

### 数据库变化

**Place 表加 3 个聚合字段**（每次新增/修改评价触发重算）：

```prisma
model Place {
  // ...现有字段...
  kidAvgScore       Float?  @map("kid_avg_score")         // 孩子评分聚合
  momAvgScore       Float?  @map("mom_avg_score")         // 妈妈评分聚合
  dadAvgScore       Float?  @map("dad_avg_score")         // 爸爸评分聚合
  reviewCount       Int     @default(0) @map("review_count")
  withChildRatingCount Int  @default(0) @map("with_child_rating_count")
  lastReviewedAt    DateTime? @map("last_reviewed_at")
  // 保留 kidScore/momScore/dadScore 作为编辑推荐分（降级用）
}
```

**重算策略**：
- 新增 `PlaceReview` → 异步（不阻塞响应）触发 `recomputePlaceScores(placeId)`
- 修改评价 → 同样重算
- 删除评价 → 同上
- 用 cron job 兜底每 6 小时全量对账一次（防止异步任务丢失）

**Parent Gender 来源**：
- auth-service 的 users 表 `gender` 字段
- travel-guide 没存 user 表，靠 `user-service.ts` 直读 SQLite
- 评价写入时把 `gender` 快照到 PlaceReview（避免 user-service 不可用时聚合失败）

## 三、跟周边便利的挂钩

### 现状
PlaceReview 表已有 5 个 bool 便利字段（hasParking / hasHighChair / hasNapRoom / strollerOk）+ kidFriendly 综合 + tags。

但**这些字段只是写死在评价记录里，没有聚合到 Place 表的「周边便利」展示**。

### 建议方案（v1.0）

**Place 表新增聚合字段**：

```prisma
model Place {
  // ...现有字段...
  // 周边便利聚合（最近 30 天评价的 bool 字段平均 true 占比）
  parkingRate       Float? @map("parking_rate")           // 0-1
  highChairRate     Float? @map("high_chair_rate")        // 0-1
  napRoomRate       Float? @map("nap_room_rate")          // 0-1
  strollerOkRate    Float? @map("stroller_ok_rate")       // 0-1
  kidFriendlyAvg    Float? @map("kid_friendly_avg")       // 1-5
  convenienceUpdatedAt DateTime? @map("convenience_updated_at")
}
```

**地点详情页「周边便利」展示**（已有板块 `place-nearby.tsx`，需增强）：

| 便利 | 展示形式 | 数据源 |
|------|----------|--------|
| 有停车场 | ✓ X% 的家庭确认 | `parkingRate` |
| 有母婴室 | ✓ X% 的家庭确认 | `napRoomRate` |
| 有高脚椅 | ✓ X% 的家庭确认 | `highChairRate` |
| 婴儿车可达 | ✓ X% 的家庭确认 | `strollerOkRate` |
| 综合亲子友好 | X.X/5 | `kidFriendlyAvg` |

**至少 3 条评价才计算**——避免噪声。

## 四、完整打分流程

### 用户路径（v1.0 上线版）

```
1. 用户在地点详情页（或攻略详情页 / 行程完成页）看到「写评价」入口
       ↓
2. 点击 → 弹窗表单
   - 必填：成人评分 1-5 星（默认 5）
   - 必填：孩子评分 1-5 星（如有孩子）+ 该次月龄
   - 可选：5 个便利 bool + 综合友好度 + 标签 + 文本
   - 可选：visitDate（v1.0 仍可选，v1.1 强制）
       ↓
3. 前端 POST /api/places/[type]/[id]/review
       ↓
4. 后端校验 + 写 PlaceReview + 异步触发 recomputePlaceScores
       ↓
5. 前端展示「评价已发布，3 维分数已更新」
       ↓
6. 用户去个人中心「我的评价」看到自己的打分记录
```

### 未登录态

- 弹窗提示「登录后评价」（主站登录，token 跨域共享）
- 已登录但 token 过期 → 401 → 引导重登

### 反作弊

当前 `lib/anti-cheat.ts` 已有规则。**确认打分模块的反作弊点**：

| 风险点 | 防护 |
|--------|------|
| 同一用户对同一地点多次打分 | PlaceReview 加 `@@unique([placeId, placeType, userId])` — **当前没有**，要加 |
| 短期集中刷分（5 分钟 10 条） | anti-cheat 检测 IP + deviceId 频次 |
| 拉踩对手（1 星 + 恶意文本） | DFA 敏感词 + 管理员后台隐藏 |
| 自吹自擂（5 星 + 营销文本） | DFA + IP 集中度检测 |
| 关联交易（自己开了攻略店，评自己店 5 星） | 检测 userId == guide.authorId |

### 未成年人保护

- 孩子的「评分」**不是孩子自己点的**——是家长代填（家长带孩子去过，记录孩子的感受）
- PlaceReview.childRating 实际语义是「家长观察到的孩子反应」
- 表单提示文案：「**孩子评分** = 您观察到孩子的反应（1-5 星）」

### 数据降级与冷启动

| 评价数 | 展示策略 |
|--------|----------|
| 0 条 | 三视角徽章隐藏，显示「暂无评分」 |
| 1-2 条 | 三视角徽章显示数字 + 「评价不足」提示 |
| ≥3 条 | 三视角徽章显示数字 + 周边便利聚合显示 |
| ≥10 条 | 上述 + 「评价分布」（1-5 星各占比） |

## 五、与现有系统的关系

### 5.1 与 `ChildRating`（行程块级）的关系
- `ChildRating`：行程块结束后的孩子感受记录（cryEpisodes / emotionalPeak / stayDurationMinutes / willingnessToReturn）—— **行程后写**，数据细颗粒度
- `PlaceReview`：地点级评价 —— **任何时候都能写**，数据粗颗粒度
- **关系**：完成 PlanRecord 时可一键「把该行程块的 ChildRating 同步转为地点评价」—— v2.0 功能

### 5.2 与 `GuideReview`（攻略评价）的关系
- `GuideReview` 是攻略的成人+孩子评价
- `PlaceReview` 是地点的成人+孩子评价
- **两套独立**：评价攻略 ≠ 评价它涉及的地点
- 攻略详情页可同时展示「对攻略的评价」+「攻略涉及的地点评价 Top 3」

### 5.3 与 `kidScore/momScore/dadScore`（种子编辑分）的关系
- 现状：Place 表 3 个字段，seed 时写死
- v1.0：保留作为「编辑推荐分」，评价 < 3 条时降级显示
- v1.5：可隐藏（不再需要，编辑推荐转做 PlaceEditorPick 标记）

### 5.4 与「勋章 / leaderboard」的关系
- 已有的 leaderboard / 徽章逻辑（如「人气攻略家」「真实记录者」）按 reviewCount 排序
- v1.0 重算 Place.aggregate 后，badge-engine 的 criteria 可以引用 reviewCount
- 「评论王」徽章 criteria 改读 PlaceReview.userId 计数

### 5.5 与「周边便利」（place-nearby.tsx）的关系
- 当前 `place-nearby.tsx` 展示 13 类周边（餐厅/酒店/医院等），是**地点周边的其他地点**
- **不要混淆**：
  - 「周边便利」= 地点附近的其他地点（已经有 13 类）
  - 「便利设施」= 这个地点自己有没有母婴室/高脚椅/停车场（这是评价聚合出来的）
- v1.0 在 place-nearby 上方新增一块「便利设施」展示 5 个 bool 的聚合率

## 六、数据契约与代码改动

### 6.1 数据库改动（migration `add_place_score_aggregation`）

```sql
ALTER TABLE spots ADD COLUMN kid_avg_score REAL;
ALTER TABLE spots ADD COLUMN mom_avg_score REAL;
ALTER TABLE spots ADD COLUMN dad_avg_score REAL;
ALTER TABLE spots ADD COLUMN review_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE spots ADD COLUMN with_child_rating_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE spots ADD COLUMN last_reviewed_at TIMESTAMP(3);
ALTER TABLE spots ADD COLUMN parking_rate REAL;
ALTER TABLE spots ADD COLUMN high_chair_rate REAL;
ALTER TABLE spots ADD COLUMN nap_room_rate REAL;
ALTER TABLE spots ADD COLUMN stroller_ok_rate REAL;
ALTER TABLE spots ADD COLUMN kid_friendly_avg REAL;
ALTER TABLE spots ADD COLUMN convenience_updated_at TIMESTAMP(3);

-- 同上应用于 restaurants / hotels / parks / playgrounds 5 张表
-- （当前 PlaceReview 用 placeId+placeType 多态，所以聚合按 placeType 分别 update 对应表）
```

或者更简单：聚合**只放 spots 表 + 一个独立 `PlaceAggregate` 视图表**，多态 key 包含 type。

**推荐**：建独立聚合表 `PlaceAggregate`：
```prisma
model PlaceAggregate {
  placeId   String @id @map("place_id")          // 复合主键的一部分
  placeType String @map("place_type")           // sight | restaurant | hotel | park | playground
  // 三视角
  kidAvgScore Float? @map("kid_avg_score")
  momAvgScore Float? @map("mom_avg_score")
  dadAvgScore Float? @map("dad_avg_score")
  reviewCount Int    @default(0) @map("review_count")
  withChildRatingCount Int @default(0) @map("with_child_rating_count")
  // 便利聚合
  parkingRate Float? @map("parking_rate")
  highChairRate Float? @map("high_chair_rate")
  napRoomRate Float? @map("nap_room_rate")
  strollerOkRate Float? @map("stroller_ok_rate")
  kidFriendlyAvg Float? @map("kid_friendly_avg")
  // 元
  lastReviewedAt DateTime? @map("last_reviewed_at")
  recomputedAt DateTime @default(now()) @map("recomputed_at")
  
  @@unique([placeId, placeType])
  @@index([placeType, reviewCount(sort: Desc)])
  @@index([placeType, kidAvgScore(sort: Desc)])
  @@map("place_aggregates")
}
```

## 七、未成年人合规底线（v1.4 第十五节 B 红线）

- **孩子评分 = 家长代评**，文案明确：「孩子评分 = 您观察到孩子的反应」
- **不收集 14 岁以下儿童账号信息**——孩子不出现在 userId
- **评价文本不强制公开儿童昵称**——表单文本字段 placeholder 提示「请勿提及孩子全名」
- **DFA 敏感词**：暴力 / 色情 / 歧视儿童（白痴/智障/脑残 等已加）
- **位置精度**：评价附位置时降精度（4 位小数 → 2 位小数）

## 九、给决策者的一句话总结

> **打分不是评价，是数据资产**——
> 一次打分产出：① 三视角分（孩子/妈妈/爸爸）② 便利设施聚合 ③ 评价文本/标签 ④ 月龄分布。
> 这些数据反哺：地点排序、行程推荐、勋章触发、AI 训练、商家合作谈判筹码。
> v1.0 先把 PlaceReview 聚合到 PlaceAggregate，让首页/详情页能展示真实评分；v1.1 再加反作弊；v1.5 接 GPS 自动打卡闭环。
